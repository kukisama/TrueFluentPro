# 作战剧本 (Ops Playbook)

> 版本：2.0 | 日期：2026-05-02
> 适用范围：TrueFluentPro 全项目 AI Agent 协作工作流

---

## 一、设计哲学

### 核心问题

1. **信箱目录混乱** — 旧系统用 `batch-0/` 编号，无语义，切换项目改目录
2. **启动摩擦高** — 需要人类先定义 goal.md、设置 active.txt、才能开始
3. **上下文断裂** — agent 不继承对话历史，却没有可靠的文件化上下文

### 设计决策

| 决策 | 原因 |
|------|------|
| **无全局注册表** | 事件目录本身就是索引，不需要额外的 manifest |
| **无 active.txt** | 人类第一句话就定义事件，agent 扫描 `.ops/` 找未完成的事件 |
| **目录名即语义** | `.ops/add-image-pipeline/` 比 `.ops/batch-47/` 可读性强 100 倍 |
| **信封即上下文** | agent 不继承对话，所有理解来自读取事件目录中的文件 |
| **第一句话即事件** | 人类对督工说的第一句话就是需求，督工立即建目录开工 |

## 二、信箱结构

```
.ops/
├── loop.ps1                     # 自动循环脚本
├── README.md                    # 本文件（简版指引）
│
├── add-image-pipeline/          # 事件目录（语义命名）
│   ├── goal.md                  # 目标+验收标准（督工第一时间创建）
│   ├── plan.md                  # 作战计划（复杂任务才有）
│   ├── round-01/
│   │   ├── order.md             # 督工 → 工匠
│   │   ├── delivery.md          # 工匠 → 督工
│   │   ├── challenge.md         # 参谋审查（可选）
│   │   └── verdict.md           # 督工裁决
│   ├── round-02/
│   │   └── ...
│   └── closure.md               # 结案 = 事件完成
│
├── fix-auth-header-bug/         # 另一个事件
│   ├── goal.md
│   └── closure.md               # 简单任务：无 plan、无 round
│
└── refactor-provider-registry/  # 另一个事件
    └── ...
```

### 2.1 事件目录命名

格式：`{动词}-{名词}-{可选修饰}`，kebab-case

```
add-image-pipeline
fix-auth-header-bug
refactor-provider-registry
enrich-tts-ssml-support
```

**防重复**：创建前检测目录是否存在。存在则追加日期后缀 `-0502` 或序号 `-v2`。

### 2.2 事件状态判断（通过文件存在性）

| 文件状态 | 含义 |
|---------|------|
| 只有 `goal.md` | 已定义目标，未开始 |
| 有 `plan.md` | 已规划轮次 |
| 有 `round-NN/order.md` | 等工匠执行 |
| 有 `round-NN/delivery.md` | 等督工/参谋审查 |
| 有 `round-NN/verdict.md` | 本轮已结（看判定是 ✅ 还是 ❌） |
| 有 `closure.md` | **事件已关闭** |

### 2.3 如何找到当前工作事件

Agent 收到"继续"时，扫描 `.ops/` 下所有子目录：
1. 排除有 `closure.md` 的（已完成）
2. 剩余的就是活跃事件
3. 如果只有一个 → 自动定位
4. 如果有多个 → 取最近修改的（或提示人类选择）
5. 如果没有 → 提示"无活跃事件"

## 三、角色定义

### 3.1 铁面督工 (Taskmaster)

| 属性 | 描述 |
|------|------|
| **文件** | `.github/agents/taskmaster.agent.md` |
| **性格** | 冷静、精确、效率至上。零信任——只信证据不信声明 |
| **权限** | read + write + search + execute（全能） |
| **决策权** | ✅ 完整 |

**核心行为**：

| 输入 | 输出 |
|------|------|
| 人类首次描述需求 | 建事件目录 + goal.md + 评估复杂度 → plan.md 或直接实现 |
| "继续" + 发现有未审查的 delivery | 审查 → verdict.md |
| "继续" + 发现需要写下一轮 order | 写 order.md |
| "继续" + 所有轮次通过 | 写 closure.md |

### 3.2 执行工匠 (Executor)

| 属性 | 描述 |
|------|------|
| **文件** | `.github/agents/executor.agent.md` |
| **性格** | 严谨、诚实、有匠心。做且仅做 order 要求的事 |
| **权限** | read + write + search + execute |
| **决策权** | ❌ 有限——只能在技术层面决策，不改需求 |

**核心行为**：

| 输入 | 输出 |
|------|------|
| "继续" + 发现有 order.md 无 delivery.md | 预审 → 实现 → 自检 → delivery.md |
| "继续" + 无待办 | "不是我的回合" |

### 3.3 毒舌参谋 (Challenger)

| 属性 | 描述 |
|------|------|
| **文件** | `.github/agents/challenger.agent.md` |
| **性格** | 尖酸刻薄、有罪推定。找最致命的弱点 |
| **权限** | read + search（**只读**） |
| **决策权** | ❌ 零——只提意见，督工可忽略 |

**核心行为**：

| 输入 | 输出 |
|------|------|
| "继续" + 发现有 delivery.md 无 challenge.md 无 verdict.md | 攻击性审查 → challenge.md |
| "继续" + 无待办 | "不是我的回合" |

## 四、工作流

### 4.1 人类的第一句话（事件创建）

```
人类对督工说："实现图片生成 Pipeline，支持 DALL-E 3 和 gpt-image-1"
                              │
                              ▼
督工：
  1. 解析需求 → 确定事件名 "add-image-pipeline"
  2. 检查 .ops/add-image-pipeline/ 是否存在
     - 不存在 → 创建
     - 存在 → 追加后缀（如 -v2）
  3. 写 goal.md（含验收标准）
  4. 评估复杂度
     - 简单 → 直接实现 → closure.md（一次性交付）
     - 复杂 → 写 plan.md → 写 round-01/order.md
```

**关键**：人类不需要自己创建目录、写 goal.md、设置任何开关。说一句话就够了。

### 4.2 路径 A：简单任务（督工一次性交付）

判断标准：≤2 文件、<100 行、无架构决策

```
人类描述需求
  → 督工建目录 + goal.md
  → 督工直接实现
  → 督工自检（编译、测试）
  → 督工写 closure.md
  → ✅ 完成
```

无 plan.md、无 round 目录。整个事件只有 `goal.md` + `closure.md`。

### 4.3 路径 B：复杂任务（三角协作）

```
人类描述需求
  → 督工建目录 + goal.md + plan.md + round-01/order.md
  → [后续全自动]

自动循环：
  "继续" → 工匠（读 order → 写 delivery）
  "继续" → [可选] 参谋（读 delivery → 写 challenge）
  "继续" → 督工（读 delivery + challenge → 写 verdict）
           ├─ ✅ 通过 → 推进到下一轮 order（或 closure）
           └─ ❌ 打回 → 下一轮 order 合并修复+新任务
  重复直到 closure.md 出现
```

### 4.4 轮次守恒

> **返工不增加轮次。修复合并到下一轮 order。**

| 原始计划 | 实际执行 |
|---------|---------|
| round-01: Provider 注册 | round-01: ✅ 通过 |
| round-02: HTTP 客户端 | round-02: ❌ 打回（序列化错误） |
| round-03: 错误处理 | round-03: **修复序列化** + 错误处理 |
| round-04: 集成测试 | round-04: 集成测试 |

总轮次 = 4，不因打回变成 5、6。

例外：合并后 > 2000 行 → 只做修复，但必须在 plan.md 标注"溢出"。

### 4.5 门禁（参谋介入时机）

在 plan.md 中用 🐍 标记哪些轮次需要参谋审查：

```markdown
| 轮次 | 目标 | 门禁 |
|------|------|------|
| round-01 | 基础结构 | — |
| round-02 | 认证+请求 | 🐍 |
| round-03 | 错误处理 | — |
| round-04 | 集成测试 | 🐍 |
```

督工也可临时决定召唤参谋（安全相关、自己有疑虑时）。

## 五、信封如何组织上下文

### 5.1 核心原理

**Agent 不继承对话历史。** 每次启动都是全新的。上下文完全来自文件。

这意味着每个文件必须**自包含**——读它的人不需要额外信息就能理解当前状态。

### 5.2 上下文链路

```
agent 收到 "继续"
  → 扫描 .ops/ 找活跃事件
  → 读 goal.md（理解最终目标）
  → 读 plan.md（理解全局规划和当前轮次）
  → 读当前 round 的文件（order/delivery/challenge/verdict）
  → 根据缺失的文件判断自己该做什么
  → 执行
```

### 5.3 每个文件的上下文责任

| 文件 | 必须包含的上下文 |
|------|----------------|
| goal.md | 最终目标 + 验收标准 + 不做什么 |
| plan.md | 全局轮次表（含状态）+ 已完成轮次的一句话总结 |
| order.md | 本轮目标 + 具体任务 + 退出标准 + 上轮打回项（如有） |
| delivery.md | 完成证据 + 编译/测试输出 + 异议（如有） |
| challenge.md | 攻击发现 + 证据 + 攻击路径 |
| verdict.md | 逐项审查结论 + 下一步指令 |

### 5.4 plan.md 的状态追踪

plan.md 是活跃事件的"进度仪表板"：

```markdown
## 轮次规划

| 轮次 | 目标 | 门禁 | 状态 | 完成日期 |
|------|------|------|------|---------|
| round-01 | 基础结构 | — | ✅ | 2026-05-02 |
| round-02 | HTTP 客户端 | 🐍 | ❌→✅ | 2026-05-03 |
| round-03 | 错误处理+序列化修复 | — | ⏳ | — |
| round-04 | 集成测试 | 🐍 | — | — |
```

每轮 verdict ✅ 后，督工更新 plan.md 对应行的状态。这样下一个 agent 启动时，读 plan.md 就知道全局进度。

## 六、文件格式

### 6.1 goal.md

```markdown
# {事件名}

> 创建日期：{YYYY-MM-DD}
> 复杂度：简单 / 复杂（N 轮）

## 目标
{一到两句话描述完成后的状态}

## 验收标准
- [ ] {可验证的条件 1}
- [ ] {可验证的条件 2}
- [ ] {可验证的条件 3}

## 背景
{为什么要做这个 + 相关资源位置}

## 不做什么
- {显式排除的范围}
```

### 6.2 plan.md

```markdown
# 作战计划：{事件名}

> 预设轮次：{N}
> 创建日期：{YYYY-MM-DD}

## 轮次规划

| 轮次 | 目标 | 预估行数 | 门禁 | 状态 | 完成日期 |
|------|------|---------|------|------|---------|
| round-01 | {目标} | ~{N} | — | ⏳ | — |
| round-02 | {目标} | ~{N} | 🐍 | — | — |

## 已完成轮次摘要
（每轮通过后追加一行，方便后续 agent 无需读历史 round 文件）

- round-01: {一句话总结做了什么}
```

### 6.3 order.md

```markdown
# Round {N} 施工单

> 事件：{event-name}
> 日期：{YYYY-MM-DD}
> 本轮目标：{一句话}

## 上轮打回修复（如有）
| 打回项 | 出处 | 修复要求 |
|--------|------|---------|

## 任务清单
- [ ] T-001: {描述}
  - 产出: {文件路径}
  - 契约: `fn xxx(a: A) -> Result<B>`
  - 关键逻辑: {步骤}
  - 测试要求: {说明}

## 退出标准
- [ ] {具体标准}

## 禁止事项
- {列表}
```

### 6.4 delivery.md

```markdown
# Round {N} 交付报告

> 事件：{event-name}
> 日期：{YYYY-MM-DD}

## ⚠️ 施工单异议（如有）
| 施工单要求 | 问题 | 我的修正 |
|-----------|------|---------|

## 任务完成
- [x] T-001: {描述} — 证据: {文件:行号}

## 编译验证
{实际输出}

## 测试验证
{实际输出}

## 变更文件
| 文件 | 操作 | 行数 |
|------|------|------|

## 已知局限
{诚实描述}
```

### 6.5 challenge.md

```markdown
# Round {N} 毒舌审查

> 事件：{event-name}
> 日期：{YYYY-MM-DD}

## 🔴 致命发现
{位置 + 攻击路径 + 后果}

## 🟡 严重质疑
{位置 + 最坏情况}

## 🟢 挑刺
{小问题}

## ✅ 无法攻破
{承认做得好的}
```

### 6.6 verdict.md

```markdown
# Round {N} 裁决

> 事件：{event-name}
> 日期：{YYYY-MM-DD}
> 判定：✅ 通过 / ❌ 打回

## 逐项审查
- ✅ T-001: {证据}
- ❌ T-002: {问题 + 修复方向}

## 参谋意见处理（如有）
| 参谋意见 | 采纳？ | 理由 |
|---------|--------|------|

## 编译/测试验证
{实际运行结果}

## 下一步
{具体指令}
```

### 6.7 closure.md

```markdown
# 结案：{事件名}

> 完成日期：{YYYY-MM-DD}
> 实际轮次：{N}（预设 {M}）

## 验收标准
- [x] {逐项确认}

## 经验教训
{什么值得记住}
```

## 七、自动发现协议

### 7.1 通用逻辑（三个 agent 共享）

```
function 找活跃事件():
    扫描 .ops/ 下所有子目录（排除 loop.ps1 等文件）
    排除有 closure.md 的目录
    如果剩余 0 个 → 返回 null
    如果剩余 1 个 → 返回该事件名
    如果剩余多个 → 取最近修改时间最新的
```

```
function 找当前轮次(事件目录):
    扫描 round-* 子目录，按数字排序
    返回最后一个（即最新的轮次）
```

### 7.2 督工的启动逻辑

```
event = 找活跃事件()

if event == null:
    → "无活跃事件。请告诉我要做什么。"（等人类输入）
    → 人类输入后：建目录 + goal.md + 评估 + 开工

round = 找当前轮次(event)

if round == null:
    → 说明 goal.md 存在但无 plan/round → 评估复杂度 → 路径 A 或 B

match round 的文件状态:
    有 order，无 delivery → "⏳ 等工匠"
    有 delivery，无 verdict → 审查 → 写 verdict
    有 verdict ✅ → 推进下一轮 order（或 closure）
    有 verdict ❌ → 写下一轮 order（含修复）
```

### 7.3 工匠的启动逻辑

```
event = 找活跃事件()
if event == null → "无活跃事件"

round = 找当前轮次(event)

match round 的文件状态:
    有 order，无 delivery → **我的回合！** 执行施工单
    其他 → "不是我的回合"
```

### 7.4 参谋的启动逻辑

```
event = 找活跃事件()
if event == null → "无活跃事件"

round = 找当前轮次(event)

match round 的文件状态:
    有 delivery，无 challenge，无 verdict → **我的回合！** 攻击性审查
    其他 → "不是我的回合"
```

## 八、循环脚本

`.ops/loop.ps1` 负责自动轮转三个 agent：

```
循环直到 closure.md 出现：
  1. 调督工 → 督工根据状态行动
  2. 如果出现了 order 无 delivery → 调工匠
  3. 如果工匠交付了 + 本轮有门禁 → 调参谋
  4. 回到步骤 1（督工审查）
```

人类只需执行 `.\.ops\loop.ps1` 然后喝咖啡。

## 九、快速启动

### 方式 1：手动逐步

```
# 对督工说第一句话（描述需求）
copilot -p "实现图片 Pipeline，支持 DALL-E 3" --agent "铁面督工" --yolo

# 然后反复说"继续"
copilot -p "继续" --agent "执行工匠" --yolo
copilot -p "继续" --agent "铁面督工" --yolo
# ... 重复
```

### 方式 2：自动循环

```powershell
# 先手动启动事件（第一句话）
copilot -p "实现图片 Pipeline，支持 DALL-E 3" --agent "铁面督工" --yolo

# 然后启动自动循环
.\.ops\loop.ps1
```

### 方式 3：极简（适合简单任务）

```
# 督工自己就能搞定
copilot -p "修复 auth header 中 Bearer 前缀重复的 bug" --agent "铁面督工" --yolo
# 如果是简单任务，督工一次性完成，无需后续循环
```
