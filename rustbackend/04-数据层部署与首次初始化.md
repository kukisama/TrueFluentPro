## 1. 数据层解耦策略

### 1.1 设计原则

> **数据库是"插座"**：系统不绑定任何特定数据库。通过 trait 抽象 + 编译期 SQL 检查，实现零业务代码改动的数据库切换。

### 1.2 支持矩阵

| 数据库 | 适用场景 | 检测方式 |
|--------|---------|---------|
| **SQLite** (默认) | 极简部署、个人使用、开发测试、<10 用户 | `DATABASE_URL` 为空或不存在 |
| **PostgreSQL** (推荐生产) | 团队使用、高并发、需要备份 | `DATABASE_URL=postgres://...` |

> **扩展性**：未来可增加 MySQL 支持，只需在 `storage` crate 中新增 `mysql.rs` 实现 `StorageBackend` trait。

### 1.3 StorageBackend trait 设计

```
trait StorageBackend: Send + Sync {
    // ─── 生命周期 ───
    async fn initialize() -> Result<()>          // 建表/迁移
    async fn health_check() -> Result<()>        // 连接检查
    async fn is_initialized() -> Result<bool>    // 是否已完成首次初始化

    // ─── 用户 ───
    async fn get_user(id) -> Result<Option<User>>
    async fn upsert_user(user) -> Result<()>
    async fn list_users(pagination) -> Result<Vec<User>>
    async fn update_user(id, changes) -> Result<()>

    // ─── Provider 与能力 ───
    async fn get_capabilities() -> Result<Vec<Capability>>
    async fn upsert_capability(cap) -> Result<()>
    async fn get_providers() -> Result<Vec<Provider>>
    async fn upsert_provider(provider) -> Result<()>
    async fn get_bindings(capability_id) -> Result<Vec<Binding>>
    async fn upsert_binding(binding) -> Result<()>

    // ─── 凭据 ───
    async fn get_credential(provider_id, key) -> Result<Option<EncryptedCredential>>
    async fn upsert_credential(cred) -> Result<()>
    async fn list_credentials(provider_id) -> Result<Vec<CredentialMeta>>

    // ─── 计费 (可选模块调用) ───
    async fn insert_usage_events(events: Vec<UsageEvent>) -> Result<()>
    async fn get_monthly_usage(user_id, resource_type, year_month) -> Result<i64>
    async fn increment_monthly_usage(user_id, resource_type, year_month, amount) -> Result<i64>
    async fn get_subscription_plan(plan_id) -> Result<Option<SubscriptionPlan>>
    async fn list_subscription_plans() -> Result<Vec<SubscriptionPlan>>
    async fn upsert_subscription_plan(plan) -> Result<()>

    // ─── 审计日志 ───
    async fn insert_audit_log(log) -> Result<()>

    // ─── 系统配置 (KV) ───
    async fn get_config(key) -> Result<Option<String>>
    async fn set_config(key, value) -> Result<()>
}
```

### 1.4 数据库选择逻辑

```
系统启动
  │
  ├─ 读取 DATABASE_URL 环境变量
  │
  ├─ 为空或不存在
  │    → 使用 SQLite
  │    → 数据文件: $DATA_DIR/truefluentpro.db
  │    → WAL 模式, 外键启用
  │    → 单文件，volume 挂载持久化
  │
  └─ 有值
       ├─ postgres:// → PostgreSQL
       └─ (未来) mysql:// → MySQL
```

### 1.5 SQLite 与 PostgreSQL 的差异处理

通过 `sqlx` 的条件编译 + SQL 迁移脚本分目录管理：

```
migrations/
├── sqlite/
│   ├── 001_initial.sql
│   ├── 002_billing.sql
│   └── ...
└── postgres/
    ├── 001_initial.sql
    ├── 002_billing.sql
    └── ...
```

| 差异点 | SQLite | PostgreSQL |
|--------|--------|-----------|
| 自增主键 | `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` |
| 时间戳 | `TEXT` (ISO 8601) | `TIMESTAMPTZ` |
| JSON | `TEXT` (手动解析) | `JSONB` (原生) |
| 二进制 | `BLOB` | `BYTEA` |
| 小数 | `REAL` | `DECIMAL(10,2)` |
| 当前时间 | `datetime('now')` | `NOW()` |
| 并发 | WAL 模式 (读不阻塞写) | MVCC (完整并发) |

> `sqlx` 的宏 `query!` 在编译期会根据目标数据库方言检查 SQL 语法，保证两套迁移脚本的正确性。

## 2. 缓存层解耦 (Redis 可选)

### 2.1 CacheBackend trait 设计

```
trait CacheBackend: Send + Sync {
    // ─── 基础操作 ───
    async fn get(key) -> Result<Option<String>>
    async fn set(key, value, ttl) -> Result<()>
    async fn delete(key) -> Result<()>
    async fn exists(key) -> Result<bool>

    // ─── 原子计数 (配额/限流) ───
    async fn increment(key, amount) -> Result<i64>
    async fn get_counter(key) -> Result<i64>
    async fn set_counter_with_ttl(key, value, ttl) -> Result<()>

    // ─── 滑动窗口限流 ───
    async fn rate_limit_check(key, max_requests, window_secs) -> Result<RateLimitResult>

    // ─── 健康检查 ───
    async fn health_check() -> Result<()>
}
```

### 2.2 三种实现

| 实现 | 检测方式 | 适用场景 | 说明 |
|------|---------|---------|------|
| **NoneCache** | `REDIS_URL` 为空 + `cache.mode = "none"` | 最极简 | 所有操作直穿 DB，计数器用 DB 原子操作 |
| **InMemoryCache** | `REDIS_URL` 为空 + `cache.mode = "memory"` (默认) | 单实例部署 | `DashMap` 做 KV，后台线程定期过期清理 |
| **RedisCache** | `REDIS_URL` 有值 | 多实例/高并发 | `redis 0.27` (tokio-comp)，连接池 |

```toml
# config/default.toml

[cache]
# "none"   — 无缓存，直穿 DB
# "memory" — 内存缓存 (DashMap), 单实例有效
# "redis"  — Redis, 多实例共享
mode = "memory"

[cache.redis]
url = ""                    # redis://user:password@host:6379/0
pool_size = 10
```

### 2.3 选择逻辑

```
系统启动
  │
  ├─ 读取 REDIS_URL 环境变量
  │
  ├─ 有值
  │    → 使用 RedisCache
  │    → 连接测试失败 → 降级到 InMemoryCache + 告警日志
  │
  └─ 为空
       ├─ config.cache.mode == "memory" → InMemoryCache (默认)
       └─ config.cache.mode == "none"   → NoneCache
```

## 3. 凭据三级链

```
请求: CredentialBroker.get("provider:tencent.tmt:secret_key")
  │
  ▼
第 1 级: 环境变量
  │  检查 env::var("TENCENT_TMT_SECRET_KEY")
  │  找到 → 返回 SecretString
  │  未找到 → 继续
  ▼
第 2 级: 数据库加密列
  │  查询 provider_credentials 表
  │  找到 → AES-256-GCM 解密 → 返回 SecretString
  │  未找到 → 继续
  ▼
第 3 级: Azure Key Vault (可选)
  │  AKV_URL 环境变量已配置?
  │  ├─ 是 → SecretClient.get_secret(key) → 返回 SecretString
  │  └─ 否 → 返回 None
  ▼
返回 None (凭据不存在)
```

### 3.1 加密方案

| 项 | 方案 |
|----|------|
| 算法 | AES-256-GCM |
| KEK (Key Encryption Key) | `MASTER_KEY_BASE64` 环境变量 (32 字节 base64) |
| Nonce | 每条凭据独立随机 12 字节，存同一行 |
| 密钥轮换 | `version` 字段，新旧并行 7 天 |
| 零泄漏保证 | `secrecy::SecretString` + `zeroize::Zeroize`；日志/trace 中永远显示 `[REDACTED]` |

### 3.2 首次启动时的 MASTER_KEY

```
首次启动
  │
  ├─ MASTER_KEY_BASE64 环境变量存在 → 使用它
  │
  └─ MASTER_KEY_BASE64 为空
       → 自动生成 32 字节随机密钥
       → 写入 $DATA_DIR/.master_key (权限 0600)
       → 终端打印警告: "自动生成的主密钥已保存到 .master_key，请妥善备份"
```

> **生产建议**：通过 Docker Compose 的 `.env` 文件显式设置 `MASTER_KEY_BASE64`。

## 4. 首次启动引导 (Setup Wizard)

### 4.1 设计目标

> 用户 `docker compose up` 后，打开浏览器（或客户端）就能看到友好的初始化向导，而不是一个需要手动写 SQL 的空系统。

### 4.2 初始化状态检测

```
GET /v1/setup/status

Response (未初始化):
{
  "initialized": false,
  "steps": [
    { "id": "database", "status": "pending", "label": "数据库初始化" },
    { "id": "admin", "status": "pending", "label": "创建管理员账户" },
    { "id": "provider", "status": "pending", "label": "配置首个 Provider" }
  ]
}

Response (已初始化):
{
  "initialized": true
}
```

### 4.3 初始化流程

```
docker compose up
       │
       ▼
网关启动
       │
       ├─ 1. 检测数据库文件/连接
       │    ├─ SQLite: 文件不存在 → 创建
       │    └─ PostgreSQL: 连接成功 → 继续
       │
       ├─ 2. 运行数据库迁移 (sqlx migrate)
       │    → 创建所有表
       │    → 插入默认套餐 (free/basic/pro/unlimited)
       │    → 插入默认能力定义 (chat/translate/speech/image/video)
       │
       ├─ 3. 检查 system_config 表的 "initialized" 键
       │    ├─ 存在且为 "true" → 正常启动
       │    └─ 不存在 → 进入 Setup 模式
       │
       ▼
  Setup 模式:
       │
       ├─ 所有端点返回 503 + JSON: { "error": "setup_required" }
       │  （除了 /v1/setup/* 和 /healthz）
       │
       ├─ 客户端检测到 "setup_required" → 显示初始化向导 UI
       │
       └─ 等待管理员完成初始化
```

### 4.4 初始化 API

```
POST /v1/setup/init
Content-Type: application/json

{
  "admin": {
    "username": "admin",            // 本地模式
    "password": "SecureP@ss123",    // 本地模式
    "display_name": "系统管理员"
  },
  "auth_mode": "local",            // "local" 或 "aad"
  "aad_config": {                   // auth_mode == "aad" 时必填
    "tenant_id": "...",
    "client_id": "...",
    "audience": "api://truefluentpro"
  },
  "provider": {                     // 可选: 首个 Provider
    "id": "azure.openai",
    "credentials": {
      "endpoint": "https://your-resource.openai.azure.com/",
      "api_key": "sk-xxxx"
    }
  },
  "billing_enabled": false          // 是否开启计费
}

Response 200:
{
  "success": true,
  "admin_token": "eyJhbG...",       // 本地模式返回首个 admin 的 JWT
  "message": "系统初始化完成，请使用此 token 登录管理后台"
}
```

### 4.5 初始化内部流程

```
POST /v1/setup/init 处理:
  │
  ├─ 1. 再次检查 initialized == false (防竞态)
  │
  ├─ 2. 创建管理员账户
  │    ├─ 本地模式: 插入 users 表 (argon2id hash password)
  │    └─ AAD 模式: 保存 AAD 配置到 system_config
  │          (首个 AAD 登录用户自动标记 admin)
  │
  ├─ 3. 配置认证模式
  │    → system_config: auth.mode = "local" / "aad"
  │
  ├─ 4. 配置首个 Provider (如果提供)
  │    → 插入 providers 表
  │    → 加密并插入 provider_credentials 表
  │    → 创建 capability_provider_bindings
  │
  ├─ 5. 配置计费开关
  │    → system_config: billing.enabled = true/false
  │
  ├─ 6. 标记初始化完成
  │    → system_config: initialized = "true"
  │
  ├─ 7. 重新加载所有模块配置
  │
  └─ 8. 返回成功 + admin token
```

### 4.6 Setup 模式的安全性

| 风险 | 缓解 |
|------|------|
| 未初始化系统被外部访问 | Setup 模式下只有 `/v1/setup/*` 和 `/healthz` 可用 |
| 初始化接口被重放 | 初始化完成后，`/v1/setup/init` 永远返回 `409 Conflict` |
| 网络窃听初始化密码 | 要求 TLS (至少自签证书) |
| 本地 Setup 页面的 CSRF | Setup API 要求 `Content-Type: application/json` |

## 5. 配置管理

### 5.1 配置加载优先级

```
高优先级
  │
  ├─ 1. 环境变量 (GATEWAY_*)          — Docker 推荐
  ├─ 2. .env 文件 (dotenvy)           — 开发用
  ├─ 3. config.toml                    — 默认值
  └─ 4. 硬编码默认值                   — 保底
低优先级
```

### 5.2 完整配置模板

```toml
# config/default.toml — 网关默认配置
# 所有值都可以通过环境变量覆盖 (GATEWAY_ 前缀 + 大写下划线分隔)
# 例如: auth.mode → GATEWAY_AUTH_MODE

# ─── 网络 ───
[server]
bind = "0.0.0.0:8080"
tls_cert = ""                     # 留空 = HTTP; 填写 = HTTPS
tls_key = ""

# ─── 认证 ───
[auth]

[auth.aad]

[auth.local]
jwt_secret = ""                   # 留空 = 首次启动自动生成
jwt_expiry_hours = 24

# ─── 数据库 ───
[database]

# ─── 缓存 ───
```

## 6. Docker 部署

### 6.1 Dockerfile

```dockerfile
# 多阶段构建 — Rust musl 静态二进制
FROM rust:1.83-alpine AS build
RUN apk add --no-cache musl-dev openssl-dev openssl-libs-static pkgconfig
WORKDIR /src
COPY . .
RUN --mount=type=cache,target=/usr/local/cargo/registry \
    --mount=type=cache,target=/src/target \
    cargo build --release --bin gateway --target x86_64-unknown-linux-musl && \
    cp target/x86_64-unknown-linux-musl/release/gateway /gateway

FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /gateway /gateway
COPY config/default.toml /app/config/default.toml
COPY migrations/ /app/migrations/
USER nonroot
WORKDIR /app
EXPOSE 8080
ENTRYPOINT ["/gateway"]
```

> 最终镜像约 **15–20 MB**，无 shell、无包管理器。

### 6.2 docker-compose.yml

```yaml
# docker-compose.yml
# 用法:
#   极简模式 (仅网关 + SQLite):  docker compose up gateway
#   标准模式 (网关 + PostgreSQL): docker compose --profile db up
#   完整模式 (网关 + PG + Redis): docker compose --profile db --profile cache up

services:
  gateway:
    image: truefluentpro/gateway:latest
    container_name: ${INSTANCE_NAME:-tfp}-gateway
    build:
      context: ./rustbackend
      dockerfile: Dockerfile
    ports:
      - "${API_PORT:-8080}:8080"
    environment:
      # ─── 数据库 (留空 = SQLite) ───
      - GATEWAY_DATABASE_URL=${DATABASE_URL:-}
      # ─── Redis (留空 = 内存缓存) ───
      - GATEWAY_CACHE_REDIS_URL=${REDIS_URL:-}
      - GATEWAY_CACHE_MODE=${CACHE_MODE:-memory}
      # ─── 认证 ───
      - GATEWAY_AUTH_MODE=${AUTH_MODE:-local}
      - GATEWAY_AUTH_AAD_TENANT_ID=${AAD_TENANT_ID:-}
      - GATEWAY_AUTH_AAD_CLIENT_ID=${AAD_CLIENT_ID:-}
      - GATEWAY_AUTH_AAD_AUDIENCE=${AAD_AUDIENCE:-api://truefluentpro}
      # ─── 凭据 ───
      - GATEWAY_CREDENTIALS_MASTER_KEY_BASE64=${MASTER_KEY_BASE64:-}
      - GATEWAY_CREDENTIALS_KEYVAULT_URL=${AKV_URL:-}
      # ─── 计费 ───
      - GATEWAY_BILLING_ENABLED=${BILLING_ENABLED:-false}
      # ─── 可观测 ───
      - GATEWAY_OBSERVABILITY_LOG_LEVEL=${LOG_LEVEL:-info}
      - RUST_LOG=${LOG_LEVEL:-info}
    volumes:
      - ${DATA_DIR:-./data}/gateway:/app/data         # SQLite + master_key
      - ${DATA_DIR:-./data}/certs:/app/certs:ro        # TLS 证书 (可选)
    depends_on:
      postgres:
        condition: service_healthy
        required: false
      redis:
        condition: service_healthy
        required: false
    restart: unless-stopped
    healthcheck:
      test: ["/gateway", "--health-check"]             # 或 curl (需要安装)
      interval: 30s
      timeout: 5s
      retries: 3

  postgres:
    image: postgres:17-alpine
    container_name: ${INSTANCE_NAME:-tfp}-postgres
    profiles: ["db"]
    ports:
      - "${PG_PORT:-5432}:5432"
    environment:
      - POSTGRES_DB=truefluentpro
      - POSTGRES_USER=tfp
      - POSTGRES_PASSWORD=${PG_PASSWORD}
    volumes:
      - ${DATA_DIR:-./data}/postgres:/var/lib/postgresql/data
    restart: unless-stopped
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U tfp -d truefluentpro"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: ${INSTANCE_NAME:-tfp}-redis
    profiles: ["cache"]
    ports:
      - "${REDIS_PORT:-6379}:6379"
    command: >
      redis-server
      --maxmemory 128mb
      --maxmemory-policy allkeys-lru
      --requirepass ${REDIS_PASSWORD:-}
    volumes:
      - ${DATA_DIR:-./data}/redis:/data
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3
```

## 7. 部署操作手册

### 7.1 极简模式首次部署

```bash
# 1. 进入部署目录
cd rustbackend/

# 2. 创建 .env
cp .env.template .env
# (极简模式只需确认 API_PORT 和 DATA_DIR)

# 3. 启动
docker compose up gateway -d

# 4. 等待健康检查通过
curl http://localhost:8080/healthz
# → "ok"

# 5. 检查初始化状态
curl http://localhost:8080/v1/setup/status
# → { "initialized": false, ... }

# 6. 执行初始化
curl -X POST http://localhost:8080/v1/setup/init \
  -H "Content-Type: application/json" \
  -d '{
    "admin": {
      "username": "admin",
      "password": "YourSecurePassword123!",
      "display_name": "管理员"
    },
    "auth_mode": "local",
    "billing_enabled": false
  }'
# → { "success": true, "admin_token": "eyJhbG..." }
```

## 8. 数据库迁移策略

### 8.1 自动迁移

```
网关启动 → StorageBackend.initialize()
  │
  ├─ sqlx migrate::run() — 执行所有未应用的迁移
  │
  ├─ 迁移成功 → 继续启动
  │
  └─ 迁移失败 → 打印错误日志 + 退出 (非零退出码)
       → Docker 自动重启 (restart: unless-stopped)
       → 管理员查看日志修复后重启
```

### 8.2 迁移脚本命名规范

```
migrations/{sqlite|postgres}/
├── 001_initial_schema.sql          # 基础表 (users, system_config)
├── 002_capabilities_providers.sql   # 能力 + Provider + 绑定表
├── 003_credentials.sql              # 加密凭据表
├── 004_billing.sql                  # 计费相关表
├── 005_audit.sql                    # 审计日志表
└── 006_default_data.sql             # 默认套餐 + 默认能力定义
```

### 8.3 版本号管理

```sql
-- sqlx 自动维护的 _sqlx_migrations 表
CREATE TABLE _sqlx_migrations (
    version BIGINT PRIMARY KEY,
    description TEXT NOT NULL,
    installed_on TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    success BOOLEAN NOT NULL,
    checksum BYTEA NOT NULL
);
```


## 10. 监控与运维

### 10.1 健康检查端点

```
GET /healthz          → 200 "ok" (进程活着就返回)
GET /readyz           → 200 / 503
{
  "status": "ready",     // "ready" / "not_ready"
  "checks": {
    "database": "ok",    // "ok" / "error: connection refused"
    "redis": "ok",       // "ok" / "not_configured" / "error: ..."
    "setup": "ok"        // "ok" / "pending" (未初始化)
  }

```

### 10.2 日志

- 结构化 JSON 日志输出到 stdout
- Docker 日志驱动收集
- 关键信息：`request_id`、`user_id`、`provider_id`、`capability_id`、`latency_ms`、`status_code`
- **永不输出**：密钥、token、密码

## 11. TLS / HTTPS

| 场景 | 方案 |
|------|------|
| 托管平台 (ACA / Railway / Fly.io) | 平台自带 TLS 终止 |
| 自建 VPS + 域名 | Let's Encrypt + certbot 自动续期 |
| 自建 VPS + 无域名 | 自签证书 (仅限内网/测试) |
| CDN (Cloudflare) | Cloudflare 提供证书 |

> 网关配置中 `server.tls_cert` 和 `server.tls_key` 填写证书路径即启用 HTTPS。

## 12. 安全检查清单

| 项 | 方案 |
|----|------|
| ✅ 传输加密 | TLS 1.2+ |
| ✅ 认证 | AAD JWT / 本地 JWT |
| ✅ 授权 | 角色 (user/admin) |
| ✅ 限流 | 按用户 ID (tower-governor + Redis/内存) |
| ✅ 配额 | 可选，月度上限 |
| ✅ 凭据安全 | AES-256-GCM 加密 + 环境变量 + KeyVault |
| ✅ 日志脱敏 | `secrecy::SecretString` + 不记录 token |
| ✅ 非 root 运行 | distroless + nonroot 用户 |
| ✅ 容器安全 | 只读文件系统 + 最小镜像 |
| ✅ 密钥轮换 | provider_credentials.version |
| ✅ 依赖审计 | `cargo deny` + `cargo audit` 在 CI 中运行 |
