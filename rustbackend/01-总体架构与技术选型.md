> **文档定位**：本文回答"这个系统是什么、由什么构成、用什么技术栈"。  
> **读者**：技术负责人、架构师、全栈工程师。  
> **前置阅读**：`backend/架构设计-多云资源中转网关.md`（模式理论）、`backend/落地设计-多云AI网关.md`（技术选型细节）。

## 1. 系统定位

> **一句话**：一个自部署的、插件化的多云 AI 能力网关（类 APIM），前端只认"我方协议"，后端以插座方式挂载多家云厂商资源，所有组件可开可关、可插可拔。

### 与传统 APIM 的区别

| 维度 | Azure APIM / Kong | 本系统 |
|------|-------------------|--------|
| 部署模式 | 云托管或自建集群 | **单二进制 + Docker Compose**，一键启动 |
| 厂商锁定 | 深度绑定平台 | **厂商无关**，Azure/腾讯/阿里/自建均为"插座" |
| 用户体系 | 依赖外部 IdP | **内建可选**：AAD 委派 / 本地账密均可 |
| 计费 | 无或平台级 | **内建可选**：管理员决定开启/关闭 |
| 数据库 | 指定 | **解耦**：SQLite / PostgreSQL / MySQL 可切换 |
| 缓存 | 指定 | **解耦**：无缓存 / Redis / 内存 可切换 |

## 2. 核心设计原则

1. **一切皆插座（Socket Design）**  
   每一个外部依赖（数据库、缓存、Provider、计费模块）都通过 trait/接口抽象，运行时根据配置决定"插上"哪个实现或完全不插。

2. **前端零感知**  
   前端（Avalonia 桌面端或未来 Web）只与"我方 Canonical Contract"对话，永远不知道后端挂了哪家厂商。

3. **管理员全权配置**  
   所有 Provider 的 Key、转换规则、能力开关、计费策略，均由管理员通过 Admin API 或首次启动向导配置，无需改代码。

4. **数据库后置**  
   系统首次启动自动检测并创建数据库；极简模式下用 SQLite（数据文件在宿主机 volume），升级时可无缝切到 PostgreSQL。

5. **横向扩展就绪**  
   通过 Redis（可选）共享配额/限流状态，API 实例可水平扩展。

## 4. 技术选型（最终方案）

### 4.1 核心栈

| 层 | 技术 | 理由 |
|----|------|------|
| **语言** | **Rust** (edition 2024) | 零 GC、内存安全、15MB 静态二进制；网关是 IO 密集场景，Rust P99 延迟最优 |
| **异步运行时** | `tokio 1.x` | 工业标准，axum/reqwest/sqlx 一致依赖 |
| **Web 框架** | `axum 0.8` + `tower` 中间件 | 类型安全路由、tower 生态完善、官方推荐 |
| **HTTP 客户端** | `reqwest 0.12` (rustls + http/2) | 调下游厂商 API |
| **序列化** | `serde` + `serde_json` | Rust 标准选择 |
| **错误处理** | `thiserror` (库) + `anyhow` (应用) | 业界惯例 |

### 4.2 认证与安全

| 需求 | 技术 | 说明 |
|------|------|------|
| AAD JWT 校验 | `jsonwebtoken 9.x` + `jwks-client-rs` | JWKS 自动缓存 5min |
| 本地账密（可选） | `argon2` + 自签 JWT | 不依赖 AAD 的极简部署 |
| 凭据加密 | `aes-gcm 0.10` + `secrecy` + `zeroize` | AES-256-GCM，密钥零泄漏 |
| KeyVault | `azure_security_keyvault_secrets` | 微软官方 Rust SDK |

### 4.3 数据层

| 需求 | 技术 | 说明 |
|------|------|------|
| 数据库 ORM | `sqlx 0.8` (编译期 SQL 检查) | 同时支持 PostgreSQL + SQLite |
| 数据库迁移 | `sqlx migrate` | 嵌入二进制，启动时自动执行 |
| 缓存 | `redis 0.27` (tokio-comp) / 内存 (`dashmap`) | Redis 可选，内存为默认 fallback |

### 4.4 可观测性

| 需求 | 技术 |
|------|------|
| 日志/追踪 | `tracing` + `tracing-subscriber` (JSON) |
| 分布式追踪 | `tracing-opentelemetry` + `opentelemetry-otlp` |
| 健康检查 | `/healthz` + `/readyz` 内置端点 |

### 4.5 配置

| 需求 | 技术 |
|------|------|
| 配置加载 | `figment` (env + TOML + CLI) |
| 环境变量 | `dotenvy` (.env 文件) |

### 4.6 部署

| 需求 | 技术 |
|------|------|
| 构建 | musl 静态链接，单二进制 ~15MB |
| 容器 | `distroless/static` 基础镜像，非 root |
| 编排 | Docker Compose (profiles 控制可选组件) |

### 4.7 "如果团队没有 Rust 经验"备选方案

> 本设计的所有架构抽象（trait → interface、crate → NuGet/module）100% 可映射到 **Go** 或 **.NET**。  
> - **Go**：`gin/echo` + `sqlc` + `go-redis` + 各家官方 Go SDK  
> - **.NET**：在现有 `backend/TrueFluentPro.Api` 上扩展，用 `ITextTranslator` 接口替代 Rust trait  
> 架构不变，只换实现语言。

## 5. 部署模式矩阵

| 模式 | 容器数 | 组件 | 适合场景 |
|------|--------|------|---------|
| **极简** | 1 | 网关 (内嵌 SQLite) | 个人/体验/开发，<10 用户 |
| **标准** | 2 | 网关 + PostgreSQL | 小团队生产，10~100 用户 |
| **完整** | 3 | 网关 + PostgreSQL + Redis | 多用户高并发，>100 用户 |

```bash
# 极简模式
docker compose up gateway

# 标准模式
docker compose --profile db up

# 完整模式
docker compose --profile db --profile cache up
```

> 所有端口、数据目录、实例名均通过环境变量配置，同一机器可跑多个实例互不冲突。

## 8. 请求处理管线

```
客户端请求
  │
  ▼
① TraceLayer         — 注入 request_id，开始 span
  │
  ▼
② CORS               — 白名单校验
  │
  ▼
③ Auth Middleware     — JWT 验签 (AAD / 本地)，提取 claims
  │                      未认证 → 401
  ▼
④ Tenant Extract     — 从 claims 提取 tenant_id、user_id、roles
  │
  ▼
⑤ Capability Gate    — 检查该用户/租户是否有权限调此能力
  │                      无权限 → 403
  ▼
⑥ Quota Check [可选] — 查配额是否超限 (Redis / 内存 / DB)
  │                      超限 → 429 + X-Quota-* headers
  ▼
⑦ Handler            — 业务处理
  │  ├─ CapabilityRouter 选 Provider
  │  ├─ CredentialBroker 取密钥
  │  ├─ Adapter 调上游，适配成 Canonical 响应
  │  └─ 返回统一格式
  ▼
⑧ Billing Record [可选] — 计量写入 (token 数 / 秒数 / 次数)
  │
  ▼
⑨ Audit Log          — 审计记录 (结构化 JSON)
  │
  ▼
响应返回客户端 (+ X-RateLimit-* + X-Quota-Remaining headers)
```
