## 二、数据库选型（SQLite vs PostgreSQL）

### 2.1 自动检测逻辑

后端通过 `database.url` 配置项自动选择数据库：

| 配置值 | 数据库 | 场景 |
|--------|--------|------|
| 空字符串（默认） | SQLite | 个人部署、开发测试 |
| `postgres://user:pass@host/db` | PostgreSQL | 团队/企业部署、高可用 |

**无需手动切换** — 两种后端实现完全相同的 `StorageBackend` trait，运行时按 URL 前缀决定。

### 2.2 简单部署（SQLite）

```bash
# 一行命令启动
docker compose up -d

# 数据文件在 ./data/truefluentpro.db
# 支持 WAL 模式，并发读性能好
# 备份：直接复制 .db 文件
```

适用场景：
- 个人使用 / 小团队（< 10 并发用户）
- 开发测试环境
- 低运维成本优先

### 2.3 高可用部署（PostgreSQL）

```bash
# 启动带 PostgreSQL 的完整栈
docker compose --profile full up -d

# 环境变量
GATEWAY_DATABASE__URL=postgres://gateway:changeme@postgres:5432/truefluentpro
GATEWAY_DATABASE__MAX_CONNECTIONS=20
```

适用场景：
- 企业部署（10+ 并发用户）
- 需要数据库读写分离、流复制
- 需要 point-in-time recovery
- 与现有 PostgreSQL 基础设施集成

### 2.4 数据迁移（SQLite → PostgreSQL）

如果用户从 SQLite 升级到 PostgreSQL：

1. 导出 SQLite 数据：
```bash
sqlite3 data/truefluentpro.db .dump > backup.sql
```

2. 修改环境变量指向 PostgreSQL
3. 启动网关 — 会自动在 PostgreSQL 中创建表结构和默认数据
4. 使用 `pgloader` 或手写脚本迁移用户数据：
```bash
pgloader sqlite:data/truefluentpro.db \
         postgresql://gateway:changeme@localhost/truefluentpro
```

### 2.5 数据恢复

| 操作 | SQLite | PostgreSQL |
|------|--------|------------|
| 备份 | 复制 .db 文件 | `pg_dump` / 流复制 |
| 恢复 | 替换 .db 文件重启 | `pg_restore` / PITR |
| 初始化 | 启动时自动创建 | 启动时自动执行 DDL |

## 三、AAD 登录集成设计

### 3.1 登录流程

```
用户点击 "使用组织账号登录"
        │
        ▼
┌───────────────────────────┐
│  MSAL.NET 设备代码流 /     │
│  交互式浏览器登录           │
│  (CloudAuthService)        │
└──────────┬────────────────┘
           │ AAD Access Token
           ▼
┌───────────────────────────┐
│  POST /v1/auth/aad-login  │
│  Header: Bearer <token>   │
│  Body: { "token": "..." } │
└──────────┬────────────────┘
           │
           ▼
┌───────────────────────────────────────────┐
│  后端验证流程：                              │
│  1. JWKS RS256 验证 AAD token 签名          │
│  2. 校验 issuer + audience                  │
│  3. 提取 oid / preferred_username / name    │
│  4. upsert_user (auth_provider = "aad")     │
│  5. 返回本网关 JWT + user info              │
└──────────┬────────────────────────────────┘
           │ Gateway JWT
           ▼
┌───────────────────────────┐
│  客户端持有 Gateway JWT    │
│  后续 API 请求携带此 token │
│  无需每次传递 AAD token    │
└───────────────────────────┘
```

### 3.2 客户端 CloudAuthService 修改

现有 `Services/Cloud/CloudAuthService.cs` 已经实现了 MSAL.NET 的 AAD 认证。需要的修改：

```csharp
// 新增：AAD 登录后换取 Gateway JWT
public async Task<GatewaySession> LoginToGatewayAsync()
{
    // 1. 获取 AAD access token（已有逻辑）
    var aadToken = await AcquireTokenAsync();

    // 2. 调用后端 /v1/auth/aad-login
    var response = await _httpClient.PostAsJsonAsync(
        $"{_gatewayUrl}/v1/auth/aad-login",
        new { token = aadToken.AccessToken }
    );

    // 3. 解析返回的 Gateway JWT + user info
    var session = await response.Content.ReadFromJsonAsync<GatewaySession>();
    return session;
}
```

### 3.3 后端 AAD 登录端点

后端已有 `/v1/auth/login`（local auth）。新增 AAD 登录端点：

```
POST /v1/auth/aad-login
Content-Type: application/json

{
    "token": "<AAD access token>"
}

Response 200:
{
    "token": "<Gateway JWT>",
    "user": {
        "id": "oid-from-aad",
        "display_name": "张三",
        "email": "zhangsan@contoso.com",
        "role": "user",
        "plan_id": "free",
        "auth_provider": "aad"
    }
}
```

## 四、资源接管设计

### 4.1 核心原则

> 无论用户是否自行配置了 AI 资源（Azure API Key 等），AAD 登录后都优先使用后端管理的资源。

```
资源优先级：
1. 后端管理的 Provider（管理员配置）  ← AAD 认证后的用户始终走这里
2. 用户本地配置（Self-Hosted 模式）  ← 未连接后端时的回退
```

### 4.2 ServiceModeManager 状态切换

```csharp
public enum ServiceMode
{
    SelfHosted,   // 本地模式：使用用户自己配置的 Azure key
    Cloud,        // Cloud 模式：通过后端 API 访问资源
}
```

**切换规则：**

| 条件 | 模式 | 说明 |
|------|------|------|
| 未配置后端 URL | SelfHosted | 现有行为不变 |
| 配置了后端 URL + 本地密码登录 | Cloud | 通过后端访问资源 |
| 配置了后端 URL + AAD 登录 | Cloud | 通过后端访问资源（优先） |
| 后端不可达 | SelfHosted（降级） | 自动降级到本地 key |

### 4.3 CloudApiClient 接管路由

现有 `Services/Cloud/CloudApiClient.cs` 需要代理以下能力路由：

| 功能 | Self-Hosted 调用 | Cloud 模式调用 |
|------|-----------------|---------------|
| Chat | AiInsightService → Azure OpenAI 直连 | CloudApiClient → `POST /v1/chat/completions` |
| TTS | AzureSpeechService.SynthesizeAsync | CloudApiClient → `POST /v1/speech/synthesize` |
| STT | AzureSpeechService.RecognizeAsync | CloudApiClient → `POST /v1/speech/transcribe` |
| 翻译 | AzureTranslatorService | CloudApiClient → `POST /v1/translate` |
| 实时翻译 | WebSocket 直连 Azure | CloudApiClient → `WSS /v1/ws/translate` |
| 图片生成 | AiInsightService → DALL-E | CloudApiClient → `POST /v1/images/generations` |
| 视频生成 | — | CloudApiClient → `POST /v1/videos/generations` |

## 五、初始化流程

### 5.1 后端初始化

```
启动 Gateway
    │
    ├─ 检测 database.url
    │   ├─ 空 → 使用 SQLite（自动创建文件）
    │   └─ postgres://... → 连接 PostgreSQL
    │
    ├─ 执行数据库迁移（CREATE TABLE IF NOT EXISTS ...）
    │
    ├─ 检查 is_initialized
    │   ├─ false → 等待 POST /v1/setup/init（创建管理员）
    │   └─ true → 正常服务
    │
    ├─ 加载 Provider 配置 → 构建 ProviderRegistry
    │
    └─ 启动 HTTP 监听
```

### 5.2 客户端首次连接

```
客户端启动
    │
    ├─ 检测 CloudSettings
    │   ├─ 无后端 URL → Self-Hosted 模式
    │   └─ 有后端 URL → 尝试 Cloud 模式
    │
    ├─ Cloud 模式：
    │   ├─ GET /healthz → 检查后端可达性
    │   │   ├─ 失败 → 降级到 Self-Hosted，显示提示
    │   │   └─ 成功 → 继续
    │   │
    │   ├─ 检查已存储的 Gateway JWT
    │   │   ├─ 有效 → 直接使用
    │   │   └─ 过期/无 → 显示登录界面
    │   │
    │   ├─ 登录方式：
    │   │   ├─ AAD 登录 → MSAL → /v1/auth/aad-login
    │   │   └─ 本地登录 → /v1/auth/login
    │   │
    │   └─ 登录成功 → 加载用户信息 + 用量
    │
    └─ Self-Hosted 模式：
        └─ 使用本地配置的 API Key（现有逻辑）
```

## 七、AAD 资源优先策略

### 7.1 配置不配置 AI 资源，都走 AAD

当用户通过 AAD 登录到后端时：

1. **有后端 Provider 配置** → 直接使用后端的 AI 资源
2. **后端无 Provider** → 返回 503，提示管理员配置
3. **用户本地有 API Key** → Cloud 模式下忽略本地 key

这意味着：
- 管理员在后端配置好 Azure OpenAI / Speech / Translator 等资源
- 用户只需 AAD 登录，不需要知道任何 API Key
- 所有用量通过后端的 billing 系统计量和限制

### 7.2 降级策略

```
正常流程：
  AAD 登录 → Gateway JWT → /v1/chat/completions → 后端 Provider → Azure

降级场景 1（后端不可达）：
  AAD 登录失败 → 回退到 Self-Hosted → 使用本地 API Key

降级场景 2（后端无 Provider）：
  AAD 登录成功 → /v1/chat/completions 返回 503 → 显示 "管理员未配置 AI 资源"

降级场景 3（配额用尽）：
  AAD 登录成功 → /v1/chat/completions 返回 429 → 显示 "本月配额已用尽"
```
