# TrueFluentPro 容器化可行性方案设计

> 本文档基于 TrueFluentPro 完整代码库架构分析，系统性评估各功能模块的容器化可行性，并设计可落地的服务端容器化方案。
> 核心目标：**降低用户端使用成本**——用户无需自行配置 Azure 订阅、API Key、Blob Storage 等，开箱即用。

---

## 四、各容器服务详细设计

### 4.1 API Gateway（api-gateway）

**职责**：统一入口、认证鉴权、请求路由

**技术选型**：ASP.NET Core + YARP（与现有 .NET 技术栈一致）

**核心特性**：

| 特性 | 说明 |
|------|------|
| TLS 终止 | Let's Encrypt 自动证书或自签证书 |
| 认证 | 简单 API Token（每个用户分配）或 JWT |
| 路由 | `/api/chat/*` → ai-chat-service, `/api/image/*` → ai-image-service, ... |
| 速率限制 | 按用户/IP 限制每分钟请求数 |
| CORS | 支持 Web 客户端跨域 |
| 健康检查 | `/health` 端点 |

**API 路由表**：

```
POST   /api/chat/completions       → ai-chat-service     (SSE streaming)
POST   /api/image/generations      → ai-image-service
POST   /api/image/edits            → ai-image-service
POST   /api/video/generations      → ai-video-service
GET    /api/video/generations/{id} → ai-video-service     (轮询状态)
GET    /api/video/generations/{id}/content → ai-video-service (下载)
POST   /api/audio/transcriptions   → ai-audio-service
POST   /api/batch/submit           → batch-worker
GET    /api/batch/{id}/status      → batch-worker
GET    /api/batch/{id}/results     → batch-worker
WS     /ws/realtime                → realtime-relay       (WebSocket)
GET    /api/models                 → ai-chat-service      (模型列表)
GET    /health                     → 健康检查
```

### 4.2 AI Chat Service（ai-chat-service）

**职责**：代理转发 Chat Completions / Responses API 请求

**对应现有代码**：`AiInsightService.cs` → 提取核心 HTTP 转发逻辑

**处理流程**：

```
客户端 POST /api/chat/completions
  ↓
  api-gateway 验证 Token
  ↓
  ai-chat-service 收到请求
  ↓
  从 Secrets 读取 Azure OpenAI API Key
  ↓
  构造上游请求 (EndpointProfileUrlBuilder 逻辑)
  ↓
  转发到 Azure OpenAI / OpenAI
  ↓
  SSE 流式响应回传给客户端
```

**关键设计**：

| 设计要素 | 方案 |
|----------|------|
| 流式响应 | SSE (Server-Sent Events) 透传，不缓冲 |
| 超时 | 上游 5 分钟超时（与现有一致） |
| 重试 | URL 候选列表自动 fallback（复用 EndpointProfileUrlBuilder） |
| 推理内容 | 透传 `reasoning_summary_text.delta` 事件 |
| 模型切换 | 请求体中 `model` 字段由客户端指定，服务端路由到对应 deployment |

### 4.3 AI Image Service（ai-image-service）

**职责**：代理图像生成 & 编辑请求

**对应现有代码**：`AiImageGenService.cs`

**处理流程**：

```
客户端 POST /api/image/generations  (JSON body: prompt, size, quality, n)
  ↓
  ai-image-service 构造上游请求
  ↓
  转发到 DALL-E endpoint
  ↓
  返回 base64 图像数据给客户端
```

**图像编辑流程**：

```
客户端 POST /api/image/edits  (multipart: image + mask + prompt)
  ↓
  ai-image-service 接收文件 → 转发到上游
  ↓
  返回编辑后的图像
```

### 4.4 AI Video Service（ai-video-service）

**职责**：视频生成编排（创建 → 轮询 → 下载）

**对应现有代码**：`AiVideoGenService.cs`

**这是容器化收益最高的模块之一**，因为：

- 视频生成是异步任务，需要长时间轮询（几分钟到十几分钟）
- 当前客户端必须保持运行来完成轮询
- 服务端可以在后台持续轮询，客户端可随时查询状态

**处理流程**：

```
客户端 POST /api/video/generations → 创建任务 → 返回 taskId
  ↓
  服务端后台轮询 Azure OpenAI Sora API
  ↓
客户端 GET /api/video/generations/{taskId} → 返回进度 (0-100%)
  ↓ (完成后)
客户端 GET /api/video/generations/{taskId}/content → 下载视频
```

**状态存储**：

| 方案 | 适用场景 |
|------|----------|
| 内存字典 + 定时清理 | 单实例小规模部署 |
| Redis | 多实例 / 需要持久化 |
| SQLite | 单实例 + 需要重启恢复 |

### 4.5 AI Audio Service（ai-audio-service）

**职责**：音频文件转写

**对应现有代码**：`AiAudioTranscriptionService.cs`

**处理流程**：

```
客户端 POST /api/audio/transcriptions (multipart: audio file + params)
  ↓
  ai-audio-service 接收音频文件
  ↓
  转发到 Whisper / gpt-4o-transcribe endpoint
  ↓
  返回转写结果 (JSON with text + segments + timestamps)
```

### 4.6 Batch Worker Service（batch-worker）

**职责**：批量转写任务编排 + Azure Blob Storage 管理

**对应现有代码**：`BlobStorageService.cs` + `BatchTranscriptionParser.cs` + `BatchPackageStateService.cs`

**容器化收益极高**：

- 用户不再需要配置 Azure Blob Storage 连接字符串
- 批量任务在服务端运行，客户端无需保持在线
- 可以在后台处理数小时的批量任务

**处理流程**：

```
客户端 POST /api/batch/submit (multipart: audio files[])
  ↓
  batch-worker 接收文件
  ↓
  上传到服务端管理的 Azure Blob Storage
  ↓
  调用 Azure Speech Batch Transcription API
  ↓
  后台轮询任务状态
  ↓
客户端 GET /api/batch/{id}/status → 返回进度
  ↓ (完成后)
客户端 GET /api/batch/{id}/results → 返回 SRT/VTT 字幕文件
```

### 4.7 Realtime Relay（realtime-relay）— P2 优先级

**职责**：WebSocket 中继，转发实时翻译音频流

**对应现有代码**：`OpenAiRealtimeTranslationService.cs`

**注意**：这是最复杂的模块，需要双向 WebSocket 中继。

```
客户端 WS → realtime-relay WS → OpenAI Realtime API
      ← (translation text)  ← (translation text)
      → (audio chunks 24kHz) → (audio chunks)
```

**中继优势**：

- 客户端不需要 OpenAI API Key
- 服务端可以做音频预处理（如服务端降噪）
- 可以在服务端加入 Phrase List 管理

**延迟风险**：额外的一跳可能增加 20-50ms 延迟，需要实测评估。

## 七、客户端改造方案

### 7.1 引入 ServerApiClient

客户端需要新增一个 `ServerApiClient` 服务，作为所有服务端通信的统一入口。

**设计原则**：
- 与现有 Service 接口兼容，可通过 DI 切换本地/远程模式
- 保持现有 UI 和 ViewModel 不变

```
                    ┌─────────────────────────┐
                    │    SettingsViewModel     │
                    │                         │
                    │  [服务模式切换]           │
                    │  ○ 本地直连 (现有模式)    │
                    │  ● 服务端模式 (新增)      │
                    │                         │
                    │  服务端地址: ________     │
                    │  访问令牌:   ________     │
                    └─────────────────────────┘
```

### 7.2 双模式架构

```csharp
// 接口不变，实现可切换
public interface IAiInsightService
{
    Task<string> GetInsightAsync(string prompt, CancellationToken ct);
}

// 现有实现：直接调用 Azure OpenAI
public class AiInsightService : IAiInsightService { ... }

// 新增实现：通过服务端代理
public class RemoteAiInsightService : IAiInsightService
{
    private readonly ServerApiClient _client;

    public async Task<string> GetInsightAsync(string prompt, CancellationToken ct)
    {
        return await _client.PostStreamAsync("/api/chat/completions", prompt, ct);
    }
}
```

## 八、安全设计

### 8.1 安全架构

```
客户端                          服务端
  │                               │
  │  HTTPS + Bearer Token         │
  ├──────────────────────────────→│
  │                               │  验证 Token (JWT / API Key)
  │                               │  检查速率限制
  │                               │  记录审计日志
  │                               │
  │                               │  从 Docker Secrets 读取
  │                               │  Azure/OpenAI API Key
  │                               │
  │                               │  转发请求到上游 AI 服务
  │                               ├───────────────→ Azure OpenAI
  │                               │
  │  返回 AI 响应                  │
  │←──────────────────────────────┤
```

### 8.2 安全要点

| 安全要素 | 方案 |
|----------|------|
| 传输加密 | HTTPS/WSS（TLS 1.3） |
| 客户端认证 | API Token（部署时生成）或 JWT |
| API Key 保护 | Docker Secrets / 环境变量，不暴露给客户端 |
| 速率限制 | 按 Token 限流，防止滥用 |
| 审计日志 | 记录所有请求的 Token、IP、时间、端点 |
| Blob SAS | 服务端生成短期 SAS URL，不暴露连接字符串 |

## 十二、风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| SSE 流式延迟增加 | 对话响应变慢 | 代理层零缓冲透传，测量端到端延迟 |
| WebSocket 中继延迟 | 实时翻译体验下降 | P2 优先级，先测量再决策；保留直连选项 |
| 视频轮询任务丢失 | 需重新生成 | 持久化任务状态到磁盘/Redis |
| API Key 泄露 | 安全风险 | Docker Secrets + 环境变量隔离 |
| 服务端宕机 | 所有用户受影响 | 客户端保留本地直连模式作为 fallback |
| 大文件上传 | 批量音频文件较大 | 分片上传 + 重试机制 |
| 多用户并发 | 资源竞争 | 速率限制 + 请求队列 |

## 十三、总结与建议

### 13.1 核心结论

1. **TrueFluentPro 的 AI 服务层完全适合容器化**——所有 HTTP 调用（Chat/Image/Video/Audio）都是无状态代理模式，直接可以提取为服务端容器。

2. **容器化的核心收益是消除用户端配置成本**——用户不再需要配置 Azure 订阅、API Key、Blob Storage 连接字符串。

3. **推荐先从单体容器开始**，验证可行性后再拆分为微服务。

4. **现有代码具有良好的可复用性**——EndpointProfileUrlBuilder、Auth 逻辑、URL 构建等核心代码可以直接复用。

5. **客户端改造侵入性小**——通过接口抽象 + DI 切换，保持现有代码结构不变。

### 13.2 优先级排序

```
P0 (立即)  → AI 代理层 (Chat/Image/Video/Audio) + API Gateway
P1 (次要)  → 客户端双模式切换 + 批量处理服务端化
P2 (延后)  → WebSocket 实时翻译中继 + Azure Speech 容器
P3 (远期)  → Kubernetes 部署 + 多租户 + 用量计费
```
