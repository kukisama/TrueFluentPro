# macOS 跨平台构建可落地方案

> **核心问题**：只有 Windows 电脑，能否用 VSCode 顺利打包 macOS 用的软件包？
>
> **结论**：**可以。** .NET SDK 原生支持从 Windows 交叉编译 `osx-arm64` / `osx-x64` 目标，配合 GitHub Actions macOS Runner 可完成完整的构建→打包→签名流水线。三个阶段均可在 Windows 上操作，每个阶段都有明确的验证标准。

---

## 方案总览

| 阶段 | 目标 | 在哪里执行 | 验证方式 |
|------|------|-----------|---------|
| **阶段一** | Windows 上交叉编译出 macOS 二进制 | Windows + VSCode | `dotnet publish -r osx-arm64` 成功产出文件 |
| **阶段二** | GitHub Actions 自动构建 macOS 包 | GitHub Cloud (macOS Runner) | Workflow 绿色，可下载 artifact |
| **阶段三** | 拷贝到 Mac 上运行 | macOS 实机 | 双击 .app 启动，UI 显示正常 |

---

|------|-----------|------|---------|

## 阶段一：Windows 交叉编译（本地 VSCode）

### 目标

在 Windows 上用 VSCode + .NET CLI，直接编译出可在 macOS 上运行的二进制文件。

### 前提条件

- Windows 10/11 + VSCode
- .NET 10 SDK 已安装（`dotnet --version` 验证）
- 项目可在 Windows 上正常编译（`dotnet build` 通过）

### 步骤 1.1：修改 .csproj 支持条件编译

**核心思路**：将 Windows 专属的 NuGet 包用 MSBuild Condition 包裹，使得 `dotnet publish -r osx-arm64` 时跳过这些包。**不改变现有 Windows 逻辑。**

在 `TrueFluentPro.csproj` 中，将 Windows 专属包移入条件组：

```xml
<!-- ═══ 跨平台依赖（所有平台共享） ═══ -->
<ItemGroup>
  <PackageReference Include="Avalonia" Version="11.3.11" />
  <PackageReference Include="Avalonia.AvaloniaEdit" Version="11.4.1" />
  <PackageReference Include="Avalonia.Desktop" Version="11.3.11" />
  <PackageReference Include="Avalonia.Themes.Fluent" Version="11.3.11" />
  <PackageReference Include="Avalonia.Fonts.Inter" Version="11.3.11" />
<!-- ═══ 跨平台依赖（所有平台共享，含 NAudio — 编译跨平台，仅 Windows 运行时可用） ═══ -->
<ItemGroup>
  <PackageReference Include="Avalonia" Version="11.3.11" />
  <PackageReference Include="Avalonia.AvaloniaEdit" Version="11.4.1" />
  <PackageReference Include="Avalonia.Desktop" Version="11.3.11" />
  <PackageReference Include="Avalonia.Themes.Fluent" Version="11.3.11" />
  <PackageReference Include="Avalonia.Fonts.Inter" Version="11.3.11" />
  <PackageReference Include="Avalonia.Diagnostics" Version="11.3.11">
    <IncludeAssets Condition="'$(Configuration)' != 'Debug'">None</IncludeAssets>
    <PrivateAssets Condition="'$(Configuration)' != 'Debug'">All</PrivateAssets>
  </PackageReference>
  <PackageReference Include="CommunityToolkit.Mvvm" Version="8.4.0" />
  <PackageReference Include="FluentAvaloniaUI" Version="2.5.0" />
  <PackageReference Include="Markdown.Avalonia" Version="11.0.2" />
  <PackageReference Include="Microsoft.CognitiveServices.Speech" Version="1.48.1" />
  <PackageReference Include="Azure.Identity" Version="1.13.2" />
  <PackageReference Include="Azure.Storage.Blobs" Version="12.22.2" />
  <PackageReference Include="Microsoft.Extensions.DependencyInjection" Version="10.0.3" />
  <PackageReference Include="NAudio" Version="2.2.1" />
  <PackageReference Include="NAudio.Lame" Version="2.1.0" />
  <PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
  <PackageReference Include="Projektanker.Icons.Avalonia.FontAwesome" Version="9.6.2" />
  <PackageReference Include="SkiaSharp" Version="2.88.9" />
</ItemGroup>

<!-- ═══ Windows 专属依赖（仅 Windows RID 时引用） ═══ -->
<ItemGroup Condition="!$(RuntimeIdentifier.StartsWith('osx')) AND !$(RuntimeIdentifier.StartsWith('linux'))">
  <PackageReference Include="Microsoft.Windows.SDK.NET" Version="10.0.18362.6-preview" />
</ItemGroup>
```

> **关键发现**：NAudio 和 NAudio.Lame 虽然是 Windows 专属功能库，但其 NuGet 包基于 .NET Standard 2.0，**编译时不依赖 Windows**，仅在运行时需要 Windows API。因此可以保留为跨平台引用——现有代码已有 `OperatingSystem.IsWindows()` 运行时保护。只有 `Microsoft.Windows.SDK.NET`（提供 `Windows.Media.Editing` 等 WinRT API）需要条件排除，因为它在编译时就需要 Windows 平台。

同时，`ApplicationManifest` 也需要条件化（`app.manifest` 是 Windows 专属格式）：

```xml
<PropertyGroup>
  <!-- app.manifest 仅在 Windows 构建时生效 -->
  <ApplicationManifest Condition="!$(RuntimeIdentifier.StartsWith('osx')) AND !$(RuntimeIdentifier.StartsWith('linux'))">app.manifest</ApplicationManifest>
</PropertyGroup>
```

### 步骤 1.2：添加编译期条件符号

在 .csproj 中添加平台条件符号，以便代码中使用 `#if` 隔离平台代码：

```xml
<PropertyGroup Condition="$(RuntimeIdentifier.StartsWith('osx'))">
  <DefineConstants>$(DefineConstants);MACOS_BUILD</DefineConstants>
</PropertyGroup>
```

> **注意**：实际代码中优先使用运行时检测 `OperatingSystem.IsMacOS()`，而不是编译期 `#if`。编译符号仅用于 P/Invoke 声明等必须在编译期隔离的场景。

### 步骤 1.3：处理 Updater 构建目标

现有的 `BuildAndCopyUpdater` Target 会构建 `Updater.exe`，这在 macOS 发布时不需要。添加条件：

```xml
<Target Name="BuildAndCopyUpdater" AfterTargets="Build"
        Condition="!$(RuntimeIdentifier.StartsWith('osx')) AND !$(RuntimeIdentifier.StartsWith('linux'))">
  <!-- 仅 Windows 构建时执行 -->
</Target>
```

### 步骤 1.4：在 VSCode 终端中执行交叉编译

```powershell
# 在 VSCode 终端 (PowerShell) 中执行

# 编译 macOS ARM64 版本（Apple Silicon: M1/M2/M3/M4）
dotnet publish TrueFluentPro.csproj -c Release -r osx-arm64 --self-contained true -o artifacts/osx-arm64

# 编译 macOS x64 版本（Intel Mac）
dotnet publish TrueFluentPro.csproj -c Release -r osx-x64 --self-contained true -o artifacts/osx-x64
```

> **为什么用 `--self-contained true`？** 目标 Mac 无需安装 .NET SDK，拷贝即用。

## 阶段二：GitHub Actions 自动构建

### 目标

配置 GitHub Actions 工作流，使用 **macOS Runner** 自动构建、打包 `.app` Bundle，生成可下载的 artifact。

### 为什么需要 GitHub Actions？

| 步骤 | Windows 本地 | GitHub Actions macOS Runner |
|------|------------|---------------------------|
| `dotnet publish` | ✅ 可以 | ✅ 可以 |
| 创建 .app 目录结构 | ✅ 可以（手动） | ✅ 自动化 |
| 设置 Unix 执行权限 `chmod +x` | ❌ 无法 | ✅ 原生支持 |
| 代码签名 `codesign` | ❌ 无法 | ✅ 有 Apple 工具链 |
| 创建 .dmg 安装镜像 | ❌ 无法 | ✅ 有 `hdiutil` |

### 步骤 2.1：创建 macOS 构建工作流

创建 `.github/workflows/build-macos.yml`：

```yaml
name: Build macOS

on:
  push:
    branches: [main]
    paths-ignore:
      - '*.md'
      - 'tools/**'
  pull_request:
    branches: [main]
  workflow_dispatch:    # 支持手动触发

jobs:
  build-macos:
    runs-on: macos-14   # Apple Silicon Runner (M1)
    strategy:
      matrix:
        rid: [osx-arm64, osx-x64]
    steps:
      - uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore
        run: dotnet restore -r ${{ matrix.rid }}

      - name: Publish
        run: |
          dotnet publish TrueFluentPro.csproj \
            -c Release \
            -r ${{ matrix.rid }} \
            --self-contained true \
            -o artifacts/publish

      - name: Create .app Bundle
        run: |
          APP="artifacts/TrueFluentPro.app"
          mkdir -p "$APP/Contents/MacOS"
          mkdir -p "$APP/Contents/Resources"

          # 复制发布产出到 .app 内部
          cp -R artifacts/publish/* "$APP/Contents/MacOS/"

          # 复制 Info.plist
          cp macOS/Info.plist "$APP/Contents/"

          # 复制图标（如果 .icns 存在）
          if [ -f "macOS/AppIcon.icns" ]; then
            cp macOS/AppIcon.icns "$APP/Contents/Resources/"
          fi

          # 设置执行权限
          chmod +x "$APP/Contents/MacOS/TrueFluentPro"

      - name: Verify .app Structure
        run: |
          echo "=== .app Bundle 结构 ==="
          find artifacts/TrueFluentPro.app -maxdepth 3 -type f | head -20
          echo "=== 主程序执行权限 ==="
          ls -la artifacts/TrueFluentPro.app/Contents/MacOS/TrueFluentPro

      - name: Create zip
        run: |
          cd artifacts
          zip -r -y TrueFluentPro-${{ matrix.rid }}.zip TrueFluentPro.app

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: macos-${{ matrix.rid }}
          path: artifacts/TrueFluentPro-${{ matrix.rid }}.zip
```

## 阶段三：拷贝到 Mac 运行验证

### 目标

### 步骤 3.1：下载与传输

1. 在 GitHub → Actions → 最新成功的 `Build macOS` run → 下载 `macos-osx-arm64` artifact
2. 解压得到 `TrueFluentPro-osx-arm64.zip`
3. 通过 U 盘 / AirDrop / 网盘传输到 Mac

### 步骤 3.2：在 Mac 上解压与运行

```bash
# 1. 解压
cd ~/Downloads
unzip TrueFluentPro-osx-arm64.zip

# 2. 移除 macOS 隔离标记（因为是从网络下载的）
xattr -cr TrueFluentPro.app

# 3. 确认执行权限
chmod +x TrueFluentPro.app/Contents/MacOS/TrueFluentPro

# 4. 首次运行（命令行启动，可以看到日志输出）
./TrueFluentPro.app/Contents/MacOS/TrueFluentPro

# 5. 或者双击运行
open TrueFluentPro.app
```

### 步骤 3.3：绕过 Gatekeeper（未签名应用）

由于没有 Apple Developer 证书签名，macOS 会阻止运行。解决方法：

**方法 A：系统设置**
1. 双击 `TrueFluentPro.app` → 提示"无法打开"
2. 打开 **系统设置 → 隐私与安全性**
3. 找到 "已阻止 TrueFluentPro" → 点击 **仍要打开**

**方法 B：命令行（推荐）**
```bash
# 移除隔离属性后直接运行
xattr -cr TrueFluentPro.app
open TrueFluentPro.app
```

## 实际文件变更清单

本方案需要修改/新增以下文件，**对现有 Windows 逻辑零影响**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `TrueFluentPro.csproj` | 修改 | 条件化 Windows SDK 引用、ApplicationManifest、Updater Target |

### 不需要修改的文件

- **所有 .cs 源码文件** — 阶段一~三不需要改代码
- **所有 .axaml 视图文件** — Avalonia UI 天然跨平台
- **app.manifest** — 保留不变，通过 .csproj 条件仅在 Windows 生效
- **release.yml** — Windows Release 工作流保持不变
- **tools/** — Updater 等工具保持不变

## 常见问题

### Q: `dotnet publish -r osx-arm64` 在 Windows 上报错怎么办？

最常见的原因是 Windows 专属 NuGet 包没有正确条件化。检查 .csproj 中 NAudio、Microsoft.Windows.SDK.NET 等包是否有 `Condition` 属性。

### Q: 为什么不用 `--self-contained false`（框架依赖模式）？

框架依赖模式要求目标 Mac 安装 .NET 10 Runtime，增加了用户门槛。自包含模式体积大约 80-100MB，但拷贝即用。

### Q: macOS 上音频功能不可用怎么办？

这是预期行为。MVP 版本的音频功能使用 Azure Speech SDK 的 `FromDefaultMicrophoneInput()` 降级模式，仅支持默认麦克风。完整音频功能需要后续集成 PortAudioSharp2。

### Q: 需要 Apple Developer 账号吗？

- **开发测试阶段**：不需要。通过 `xattr -cr` 和手动允许即可运行未签名应用。
- **正式分发**：建议购买 Apple Developer Program ($99/年)，用于代码签名和公证，避免用户收到安全警告。

### Q: GitHub Actions macOS Runner 收费吗？

- 公开仓库：**免费**
- 私有仓库：每月 2000 分钟免费额度（macOS 消耗 10x，相当于 200 分钟 macOS 构建时间）
