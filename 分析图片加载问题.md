# 图片加载性能分析报告

## 一、核心痛点

用户期望：**切换到会话后，尽快看到第一张图片，其余图片在右侧列表中渐进加载即可**。

实际体验：**首张图片出现延迟明显（100–500ms），UI 线程有明显卡顿感**。

---

## 二、当前架构全景

### 2.1 关键文件与职责

| 文件 | 职责 |
|------|------|
| `Controls/ViewportLazyImage.cs` | 视口感知的懒加载图片控件 |
| `Controls/FilePathToBitmapConverter.cs` | LRU 位图缓存 + 解码 |
| `Controls/CachedContentControl.cs` | 会话视觉树缓存（LRU 驱逐） |
| `ViewModels/MediaStudioViewModel.cs` | 会话管理、异步加载调度 |
| `ViewModels/MediaSessionViewModel.cs` | 两阶段消息渲染、消息高度估算 |
| `Views/MediaStudioView.axaml.cs` | Scroll-to-bottom、第二批消息刷入 |
| `Views/MediaStudioView.axaml` | XAML 布局，`StackPanel` 全量渲染 |

### 2.2 完整加载时序

```
用户点击切换会话
  │
  ├─ ① CurrentSession setter → EnsureSessionLoadedAsync()
  │     └─ Task.Run() → LoadSessionData() 【后台线程，SQLite bundle 查询】
  │         └─ 返回 (messages, tasks) 对象列表
  │
  ├─ ② Dispatcher.UIThread.InvokeAsync() → ApplySessionData()
  │     └─ session.ReplaceAllMessages(messages, viewportHeight)
  │         ├─ 计算首批条数 (MinFirstPaintCount=2 ~ MaxFirstPaintCount=15)
  │         ├─ 只将尾部 firstBatch 条加入 Messages 集合
  │         └─ 剩余标记为 _pendingSecondBatch
  │
  ├─ ③ UI 布局通道（DispatcherPriority.Loaded）
  │     ├─ ListBox 用 StackPanel 全量渲染所有 Messages 项
  │     ├─ 每个消息中的 MediaPaths 创建 ViewportLazyImage 控件
  │     └─ ViewportLazyImage.OnFilePathChanged() 触发
  │         ├─ 🟢 LRU 缓存命中 → 立即显示（微秒级）
  │         ├─ 🟢 PNG 头读取 → 锁定占位高度（微秒级）
  │         └─ 🔴 缓存未命中 → Source=null，注册 LayoutUpdated/ScrollChanged 监听
  │
  ├─ ④ ScrollToBottomReliable() → 触发 scroll 事件
  │     └─ ViewportLazyImage.TryLoadWhenVisible()
  │         ├─ 检查 IsEffectivelyVisible + 视口范围
  │         └─ 🔴🔴🔴 FilePathToBitmapConverter.TryGetBitmap(FilePath)
  │             ├─ File.Exists() 检查
  │             ├─ lock (_lruList) 加锁
  │             ├─ LRU 缓存未命中
  │             ├─ DecodeThumbnail()
  │             │   ├─ File.OpenRead() ← 同步磁盘 I/O
  │             │   └─ Bitmap.DecodeToWidth(stream, 400) ← 同步 CPU 解码
  │             └─ 返回 Bitmap → Source = bitmap
  │
  └─ ⑤ FlushPendingMessagesDeferred()（Background 优先级）
        └─ 插入第二批消息到 Messages 头部 + 补偿滚动位置
```

## 三、根因分析

### 🔴 根因 #1：位图解码在 UI 线程同步执行

**文件**: `Controls/ViewportLazyImage.cs` 第 173 行

```csharp
// TryLoadWhenVisible() 中 — 此处在 UI 线程执行
Bitmap? bitmap = FilePathToBitmapConverter.TryGetBitmap(FilePath);
```

**文件**: `Controls/FilePathToBitmapConverter.cs` 第 169–174 行

```csharp
private static Bitmap? DecodeThumbnail(string filePath)
{
    using var stream = File.OpenRead(filePath);       // ← 同步磁盘 I/O
    return Bitmap.DecodeToWidth(stream, ThumbnailMaxPixels); // ← 同步 CPU 解码
}
```

**影响**: 对于典型的 AI 生成图片（1024×1024 PNG），单次解码耗时 **50–200ms**。在 UI 线程执行意味着渲染管线完全阻塞，用户看到空白占位后需等待解码完成才能看到图片。

**为什么第一张图特别慢**: 首次打开会话时 LRU 缓存为空（冷启动），100% 触发 DecodeThumbnail，且解码期间 UI 完全冻结。

### 🔴 根因 #2：`OnFilePathChanged` 中也调用了 TryGetBitmap

**文件**: `Controls/ViewportLazyImage.cs` 第 87 行

```csharp
private void OnFilePathChanged()
{
    // ...
    // ① 尝试 LRU bitmap 缓存直接命中（虚拟化回收最常见路径）
    var cached = FilePathToBitmapConverter.TryGetBitmap(path);
```

这里意图是"快速检查缓存"，但 `TryGetBitmap` 的实现是**缓存未命中时直接解码**（而不是仅检查缓存）。当 `OnFilePathChanged` 在消息批量添加时连续触发，**每个图片都在 UI 线程尝试完整解码**，哪怕它们不在视口内也会走一遍。

**这是最严重的问题**：本应是"检查缓存 → 设占位 → 等视口内再加载"的流程，却因为 `TryGetBitmap` 在缓存 miss 时会直接解码，变成了"所有图片在 FilePathChanged 时就同步解码"。

### 🟡 根因 #3：StackPanel 全量布局

**文件**: `Views/MediaStudioView.axaml` 第 312–315 行

```xml
<ListBox.ItemsPanel>
    <ItemsPanelTemplate>
        <StackPanel/>
    </ItemsPanelTemplate>
</ListBox.ItemsPanel>
```

使用 `StackPanel` 而非 `VirtualizingStackPanel`，意味着所有消息项**全部实例化**（虽然两阶段渲染限制了数量，但首批 2–15 条中的所有图片都会触发 `OnFilePathChanged`）。

### 🟡 根因 #4：两阶段渲染增加帧延迟

两阶段渲染本身是好设计，但：

1. 首批通过 `DispatcherPriority.Loaded` 调度 scroll-to-bottom
2. 第二批通过 `DispatcherPriority.Background` 延迟插入
3. 滚动补偿再用 `DispatcherPriority.Render` 调整

总共跨越 **3+ 个 Dispatcher 优先级**，叠加同步解码延迟，导致首张图片显示需要等多个帧周期。

## 四、现有优化（值得保留的设计）

| 优化 | 位置 | 评价 |
|------|------|------|
| ✅ LRU 位图缓存（200 条） | `FilePathToBitmapConverter` | 二次访问秒开 |
| ✅ PNG 头 24 字节读取尺寸 | `ReadPngHeaderDimensions` | 微秒级占位高度锁定 |
| ✅ 尺寸永久缓存 | `_dimensionCache` | 防止虚拟化回收后高度塌缩 |
| ✅ 视口范围检测 + 160px 预加载 | `ViewportLazyImage.TryLoadWhenVisible` | 按需加载思路正确 |
| ✅ 会话视觉树缓存 | `CachedContentControl` | 切换已访问过的会话秒开 |
| ✅ 后台 SQLite 查询 | `LoadSessionFromSqliteAsync` | DB I/O 不阻塞 UI |
| ✅ Bundle 查询替代 N+1 | `GetSessionBundle` | 减少 DB 往返 |

## 五、优化方案

### 方案概览

```
优先级    方案                           预期效果            改动量
─────────────────────────────────────────────────────────
P0     拆分 TryGetBitmap 为查询+解码      消除 UI 线程阻塞     小
P0     异步化 DecodeThumbnail             首图 <50ms 显示      中
P1     预热首批可见图片缓存                冷启动零等待          小
P2     渐进式加载反馈                      感知延迟降低          小
P3     考虑 VirtualizingStackPanel         减少不可见项实例化    中
```

### P0：拆分缓存查询与解码逻辑（关键修复）

**问题**: `TryGetBitmap` 既是缓存查询又是解码器，导致在 `OnFilePathChanged` 中调用时会意外触发同步解码。

**方案**: 新增 `TryGetCachedBitmap` 方法，**仅查询缓存**，不执行解码。

```
FilePathToBitmapConverter 修改:
├─ TryGetCachedBitmap(path)   // 新增：纯缓存查询，O(1)，无 I/O
│   └─ lock → 查 _lruMap → 命中返回 Bitmap，未命中返回 null
│
└─ TryGetBitmap(path)         // 保持不变：缓存 + 解码（仅在后台线程调用）

ViewportLazyImage 修改:
├─ OnFilePathChanged()
│   └─ 改用 TryGetCachedBitmap(path) ← 仅缓存查询，不触发解码
│
└─ TryLoadWhenVisible()
    └─ 改为异步执行 TryGetBitmap（见 P0 第二部分）
```

### P0：异步化位图解码（关键修复）

**方案**: 将 `TryLoadWhenVisible` 中的解码操作移到后台线程。

```
当前流程（UI 线程阻塞）:
TryLoadWhenVisible()
  → TryGetBitmap() → DecodeThumbnail() → [阻塞 50-200ms]
  → Source = bitmap

优化后流程（UI 线程不阻塞）:
TryLoadWhenVisible()
  → _isLoading = true（防重入）
  → Task.Run(() => TryGetBitmap())  ← 后台解码
  → await → Dispatcher.UIThread.InvokeAsync(() => Source = bitmap)

伪代码:
private async void TryLoadWhenVisible()
{
    if (_bitmapLoaded || _isLoading || string.IsNullOrWhiteSpace(FilePath))
        return;

    if (!IsEffectivelyVisible || !IsInViewport())
        return;

    _isLoading = true;
    var path = FilePath;

    // 后台线程解码
    var bitmap = await Task.Run(() => FilePathToBitmapConverter.TryGetBitmap(path));

    if (bitmap != null && FilePath == path)  // 防止路径已变
    {
        ApplyFixedSize(bitmap.PixelSize);
        Source = bitmap;
        _bitmapLoaded = true;
        UnsubscribeEvents();
    }
    _isLoading = false;
}
```

**注意事项**:
- 需要添加 `_isLoading` 标志防止 LayoutUpdated/ScrollChanged 触发重复加载
- 需要在 `await` 后检查 `FilePath` 是否已变（用户可能快速切换会话）
- `FilePathToBitmapConverter.TryGetBitmap` 内部的 `lock (_lruList)` 已保证线程安全

### P1：预热首批可见图片缓存

**时机**: 在 `LoadSessionFromSqliteAsync` 的后台线程中（DB 查询完成后、回到 UI 线程前），提前解码首批消息的图片。

```
当前流程:
  Task.Run(() => LoadSessionData())  // 后台: DB 查询
  → UI 线程: ApplySessionData()      // 消息添加到 UI
  → ViewportLazyImage 逐个解码       // 此时才开始

优化后:
  Task.Run(() => {
      var data = LoadSessionData();
      // 🆕 在后台线程预热最后几条消息的图片缓存
      PreWarmImageCache(data.Messages);
  })
  → UI 线程: ApplySessionData()
  → ViewportLazyImage 命中缓存 ← 因为已预热，直接显示！

预热逻辑:
private void PreWarmImageCache(List<ChatMessageViewModel> messages)
{
    // 从尾部取最后 3-5 条消息（用户最先看到的）
    foreach (var msg in messages.TakeLast(5))
    {
        foreach (var path in msg.MediaPaths)
        {
            FilePathToBitmapConverter.TryGetBitmap(path); // 后台解码并缓存
        }
    }
}
```

**效果**: 用户切换会话时，首批消息的图片已在缓存中，`OnFilePathChanged` 中的 `TryGetCachedBitmap` 直接命中，**零等待显示**。

### P2：加载态反馈

当前图片未加载时显示为空白占位区（Source=null 但 Width/Height 已锁定）。用户看到一个空白方块不知道是否在加载。

**方案**: 在 `ViewportLazyImage` 中添加轻量级加载指示：

- 方案 A：设置一个灰色/半透明的占位背景色，解码完成后移除
- 方案 B：使用一个简单的 Loading 文字覆盖

这是**感知性能优化**——即使实际加载时间不变，用户也能理解"正在加载"而非"出了问题"。

### P3：评估 VirtualizingStackPanel

当前使用 `StackPanel` 全量渲染，原因是注释中提到"避免 Extent 震荡"。但由于：
- PNG 头读取已能提供精确的占位高度
- `_dimensionCache` 永久保存尺寸

已有足够基础支持 `VirtualizingStackPanel`。可评估切换后是否仍有震荡，如果稳定则能显著减少不可见消息的控件实例化开销。

## 六、方案实施优先级与风险评估

### 推荐实施顺序

```
第一步（P0，必做）:
  1. 新增 TryGetCachedBitmap — 纯缓存查询
  2. OnFilePathChanged 改用 TryGetCachedBitmap
  3. TryLoadWhenVisible 异步化
  
  ✅ 这一步完成后，UI 线程不再有任何位图解码操作
  ⚡ 预期首图延迟：从 100-500ms → 20-50ms

第二步（P1，推荐）:
  4. 后台预热首批图片缓存
  
  ✅ 冷启动首图延迟进一步降至接近 0ms
  ⚡ 用户切换会话时图片"秒出"

第三步（P2，可选）:
  5. 加载占位反馈
  
  ✅ 改善感知体验

第四步（P3，慎重评估）:
  6. VirtualizingStackPanel 切换
  
  ⚠️ 需要充分测试虚拟化稳定性
```

### 风险点

| 风险 | 应对措施 |
|------|----------|
| 异步解码后 FilePath 已变 | `await` 后检查 `FilePath == path` |
| 多个 TryLoadWhenVisible 并发 | 添加 `_isLoading` 防重入标志 |
| 后台线程解码 + lock 竞争 | 现有 `lock (_lruList)` 已保证安全 |
| 预热过多图片浪费 CPU | 限制预热数量（5 条），可配置 |
| VirtualizingStackPanel 回退 | 保留 StackPanel 配置作为 fallback |

## 七、核心结论

**根本问题不是懒加载不彻底，而是懒加载的"最后一英里"（实际解码）在 UI 线程同步执行**。

现有架构在很多层面做得很好（LRU 缓存、PNG 头读取、视口检测、会话缓存），但所有这些优化都在解码环节"功亏一篑"——因为 `DecodeThumbnail` 的 `File.OpenRead` + `Bitmap.DecodeToWidth` 是同步的，且直接在 UI 线程调用。

更严重的是，`OnFilePathChanged` 本应只查缓存的地方也调用了 `TryGetBitmap`（会触发解码），使得**即使不在视口内的图片也会被同步解码**。

修复方案的核心思路：
1. **分离关注点**：缓存查询（同步、UI 线程安全） vs 解码操作（异步、后台线程）
2. **提前预热**：利用已有的后台线程（DB 查询线程），顺手预解码首批可见图片
3. **异步化最后一英里**：视口内图片的解码也不阻塞 UI 线程

三步优化后，理论上首图延迟可从 **100–500ms 降至接近 0ms**。
