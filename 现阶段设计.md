### 3.1 皇帝类（God Class）

#### MainWindowViewModel — 1 139 行 / 4 个 partial 文件

| 文件 | 行数 | 职责 |
|------|------|------|
| `MainWindowViewModel.cs` | 399 | 构造函数、配置加载、子 VM 创建、崩溃上下文 |
| `MainWindowViewModel.Commands.cs` | 25 | Command/VM 属性声明 |
| `MainWindowViewModel.Core.cs` | 92 | 窗口生命周期、PostShow 初始化、Dispose |
| `MainWindowViewModel.TranslationAndUi.cs` | 623 | 翻译服务、UI 状态、事件处理、悬浮字幕 |

**问题指标：**

- 持有 **6 个子 ViewModel** + **9 个 Service/字段依赖**
- 承担 **12 项不同职责**：翻译编排、UI 状态、配置、音频播放、文件库、批处理、音频设备、悬浮字幕、AI 洞察、历史管理、应用生命周期、Command 执行
- 用 partial class 分文件只是物理拆分，**逻辑耦合度未降低**
- 构造函数一次性 `new` 所有子 ViewModel 并通过 lambda 回调互联

#### SpeechTranslationService — 1 255 行

- 混合了音频捕获、语音识别、翻译协调、字幕写入、会话日志、音频电平监控、诊断发布等
- 构造函数中执行文件 I/O（`InitializeSessionFile`）
- 60+ 私有字段

#### SettingsViewModel — 1 265 行（经两轮精简，原 1 869 行）

已替代原 `ConfigCenterView.axaml.cs`（已删除）的全部配置编辑逻辑，包括：终结点 CRUD、订阅管理、存储验证、AI 认证检测、模型刷新、500ms 防抖自动保存。已注册为 DI Singleton。
经 `Set<T>` 提取、注释删除、属性压缩等两轮精简后从 1 869 行降至 1 265 行（-32.3%）。
图像/视频参数已改为代理模式，直接绑定共享 `MediaGenConfig` 实例（见 3.9 节）。

### 3.2 依赖注入（DI）不完整

#### 现状：仅 4 个服务注册

```csharp
// App.axaml.cs
services.AddSingleton<ConfigurationService>();
services.AddSingleton<AzureSubscriptionValidator>();
services.AddSingleton<SettingsViewModel>();
services.AddSingleton<MainWindowViewModel>();
```

其余 **10+ ViewModel 和全部 Service** 均在 `MainWindowViewModel` 构造函数中手动 `new`：

```csharp
// MainWindowViewModel.cs:74-75
var azureTokenProvider = new AzureTokenProvider("ai");
_aiInsightService = new AiInsightService(azureTokenProvider);

// MainWindowViewModel.cs:80-159
ConfigVM = new ConfigViewModel(configService, subscriptionValidator, _config, ...);
Playback = new PlaybackViewModel(...);
FileLibrary = new FileLibraryViewModel(...);
BatchProcessing = new BatchProcessingViewModel(...);
AudioDevices = new AudioDevicesViewModel(...);
AiInsight = new AiInsightViewModel(...);
```

#### Service Locator 反模式

```csharp
// App.axaml.cs:14
public static IServiceProvider Services { get; private set; } = null!;
```

全局静态 `App.Services` 可在任意位置访问，构成 Service Locator 反模式。

#### 回调委托替代注入

子 ViewModel 的依赖不通过接口注入，而是用大量 `Func<>`/`Action<>` 回调：

```csharp
// ConfigViewModel 构造函数接收 5 个委托
ConfigVM = new ConfigViewModel(
    configService, subscriptionValidator, _config,
    () => { /* 刷新命令 */ },
    cfg => _translationService?.UpdateConfigAsync(cfg),
    (eventName, message, isSuccess) => AppLogService.Instance.LogAudit(...),
    () => BatchProcessing?.IsReviewSummaryLoading ?? false,
    () => _mainWindow);
```

### 3.3 零接口 — 违反依赖倒置原则

**全项目未定义任何 `interface`。** 所有 Service 和 ViewModel 都是具体类，导致：

- 单元测试无法 Mock
- 高层模块直接依赖底层实现
- 替换实现需要修改所有调用点

### 3.4 静态单例滥用

| 类 | 访问方式 | 被引用处 | 问题 |
|----|---------|---------|------|
| `CrashLogger` | `static class`，所有方法静态 | 全局 15+ 处 | 可变状态（`_breadcrumbs`, `_contextProvider`），不可 mock；`AddBreadcrumb`/`SetContextProvider` 已加 `[Conditional("DEBUG")]`，Release 零开销 |
| `PathManager` | `Lazy<PathManager>` + `Instance` | 全局 10+ 处 | `SessionsPath` 可变，被 ConfigurationService 等间接依赖 |
| `AppLogService` | 手动 `Initialize()` + `Instance` | 全局 10+ 处 | 非线程安全初始化，隐式依赖 |

这三个 static/singleton 通过 `PathManager.Instance`、`AppLogService.Instance`、`CrashLogger.AddBreadcrumb()` 在 View、ViewModel、Service 层被广泛直接调用，形成**隐式全局依赖**。

### 3.7 无测试基础设施

- 无 xUnit/NUnit/MSTest 项目
- 无任何自动化测试
- 由于 0 个接口 + 静态依赖 + 构造函数副作用，当前代码几乎无法编写单元测试

### 3.9 配置参数同步架构（已改进）

图像/视频 8 个生成参数（ImageSize、ImageQuality、ImageFormat、ImageCount、VideoAspectRatio、VideoResolution、VideoSeconds、VideoVariants）已实现**单一数据源 + 双向实时同步**：

```
MediaGenConfig : ObservableObject  ← 唯一数据源（8 个 SetProperty 属性）
       │
       ├── SettingsViewModel（代理属性，get/set 直通 MediaGenConfig）
       │     └── PropertyChanged → OnMediaGenConfigPropertyChanged
       │           → OnPropertyChanged（刷新 UI）
       │           → MarkDirty（500ms 防抖 → SaveConfigAsync）
       │           → SyncVideoDimensionsFromSelection（宽高派生）
       │
       └── MediaSessionViewModel（暴露 GenConfig 属性供 View 绑定）
             └── PropertyChanged → ReevaluateReferenceImageValidation
```

**关键设计点：**
- 两端共享同一个 `MediaGenConfig` 实例引用，修改天然双向可见
- `_suppressDirty` 标志防止 `ApplyToConfig` 中 Clamp 校正触发二次保存
- `SetProperty` 内置相等性检查，值不变不触发事件
- `MediaGenSession` 不再保存会话级 Override —— 所有会话共享全局参数
- XAML 绑定路径：SettingsView 用 `{Binding Settings.ImageSize}`，MediaStudioView 用 `{Binding GenConfig.ImageSize}`

### 3.8 自定义 RelayCommand 冗余

项目自行实现了 `RelayCommand`（`ViewModels/RelayCommand.cs`），但已引用 `CommunityToolkit.Mvvm 8.4.0`，该库提供了功能更完善的 `RelayCommand` 和 `AsyncRelayCommand`（含自动 `CanExecuteChanged`、异常传播、取消支持等）。

## 六、当前架构优点

在指出问题的同时，也应认可以下做法：

1. **MVVM 基础骨架已建立** — 使用 CommunityToolkit.Mvvm 的 `ObservableObject`，ViewModelBase 继承链清晰
2. **NavigationView Shell 模式** — FluentAvalonia NavigationView + Panel/IsVisible 页面切换，符合 WinUI 惯例
3. **CrashLogger 诊断能力** — 崩溃日志包含 GC/内存/线程池/上下文快照/面包屑，信息量充足。`AddBreadcrumb`/`SetContextProvider` 已加 `[Conditional("DEBUG")]`，Release 下仅保留崩溃捕获（Init/Write/HookAvaloniaUiThread），面包屑零开销
4. **PostShow 初始化机制** — `RegisterPostShowInitializationAction` 延迟到窗口显示+配置加载后执行，避免冷启动阻塞
5. **InlineListContextMenuController** — 右键菜单已抽象为可复用控件
6. **配置保存防抖机制** — SettingsViewModel 500ms 防抖自动保存，避免频繁写盘
7. **CommunityToolkit.Mvvm 已引入** — 为后续迁移到 `[ObservableProperty]`/`[RelayCommand]` 提供了条件
8. **ConfigCenterView → SettingsViewModel 迁移已完成** — 1 317 行 View code-behind 业务逻辑已成功迁移至 ViewModel 层，符合 MVVM 职责划分
9. **MediaStudioWindow/View 合并已完成** — 消除了两套近乎相同的 code-behind 实现
10. **MediaGenConfig → ObservableObject + 单一数据源** — 图像/视频 8 个参数由 `MediaGenConfig` 统一管理，Settings 和 Media Studio 双向实时同步，消除了三份状态副本问题
