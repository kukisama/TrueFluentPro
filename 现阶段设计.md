## 三、已识别的架构问题

### 3.1 皇帝类（God Class）

#### MainWindowViewModel — 1 153 行 / 4 个 partial 文件

| 文件 | 行数 | 职责 |
|------|------|------|
| `MainWindowViewModel.cs` | 383 | 构造函数、配置加载、子 VM 创建、崩溃上下文 |
| `MainWindowViewModel.Commands.cs` | 25 | Command/VM 属性声明 |
| `MainWindowViewModel.Core.cs` | 92 | 窗口生命周期、PostShow 初始化、Dispose |
| `MainWindowViewModel.TranslationAndUi.cs` | 653 | 翻译服务、UI 状态、事件处理、悬浮字幕 |

**问题指标：**

- 持有 **6 个子 ViewModel** + **9 个 Service/字段依赖**
- 承担 **12 项不同职责**：翻译编排、UI 状态、配置、音频播放、文件库、批处理、音频设备、悬浮字幕、AI 洞察、历史管理、应用生命周期、Command 执行
- 用 partial class 分文件只是物理拆分，**逻辑耦合度未降低**
- 构造函数一次性 `new` 所有子 ViewModel 并通过 lambda 回调互联

#### SpeechTranslationService — 1 255 行

- 混合了音频捕获、语音识别、翻译协调、字幕写入、会话日志、音频电平监控、诊断发布等
- 构造函数中执行文件 I/O（`InitializeSessionFile`）
- 60+ 私有字段

#### 其他大文件

| 文件 | 行数 | 备注 |
|------|------|------|
| `BatchProcessingViewModel.cs` | 2 216 | 复盘全流程，含批量 API 调用 |
| `MediaSessionViewModel.cs` | 1 642 | 单个媒体会话管理 |
| `ConfigCenterView.axaml.cs` | 1 317 | View code-behind 包含大量业务逻辑 |
| `MediaStudioWindow.axaml.cs` | 1 286 | 同上，且与 MediaStudioView.axaml.cs 近乎重复 |
| `MediaStudioView.axaml.cs` | 1 277 | 嵌入版本，代码与 Window 版本高度重复 |

### 3.2 依赖注入（DI）不完整

#### 现状：仅 3 个服务注册

```csharp
// App.axaml.cs
services.AddSingleton<ConfigurationService>();
services.AddSingleton<AzureSubscriptionValidator>();
services.AddSingleton<MainWindowViewModel>();
```

其余 **10+ ViewModel 和全部 Service** 均在 `MainWindowViewModel` 构造函数中手动 `new`：

```csharp
// MainWindowViewModel.cs:74-75
var azureTokenProvider = new AzureTokenProvider("ai");
_aiInsightService = new AiInsightService(azureTokenProvider);

// MainWindowViewModel.cs:80-159
ConfigVM = new ConfigViewModel(configService, subscriptionValidator, _config, ...);
Playback = new PlaybackViewModel(...);
FileLibrary = new FileLibraryViewModel(...);
BatchProcessing = new BatchProcessingViewModel(...);
AudioDevices = new AudioDevicesViewModel(...);
AiInsight = new AiInsightViewModel(...);
```

#### Service Locator 反模式

```csharp
// App.axaml.cs:14
public static IServiceProvider Services { get; private set; } = null!;
```

全局静态 `App.Services` 可在任意位置访问，构成 Service Locator 反模式。

#### 回调委托替代注入

子 ViewModel 的依赖不通过接口注入，而是用大量 `Func<>`/`Action<>` 回调：

```csharp
// ConfigViewModel 构造函数接收 5 个委托
ConfigVM = new ConfigViewModel(
    configService, subscriptionValidator, _config,
    () => { /* 刷新命令 */ },
    cfg => _translationService?.UpdateConfigAsync(cfg),
    (eventName, message, isSuccess) => AppLogService.Instance.LogAudit(...),
    () => BatchProcessing?.IsReviewSummaryLoading ?? false,
    () => _mainWindow);
```

### 3.3 零接口 — 违反依赖倒置原则

**全项目未定义任何 `interface`。** 所有 Service 和 ViewModel 都是具体类，导致：

- 单元测试无法 Mock
- 高层模块直接依赖底层实现
- 替换实现需要修改所有调用点

### 3.4 静态单例滥用

| 类 | 访问方式 | 被引用处 | 问题 |
|----|---------|---------|------|
| `CrashLogger` | `static class`，所有方法静态 | 全局 15+ 处 | 可变状态（`_breadcrumbs`, `_contextProvider`），不可 mock |
| `PathManager` | `Lazy<PathManager>` + `Instance` | 全局 10+ 处 | `SessionsPath` 可变，被 ConfigurationService 等间接依赖 |
| `AppLogService` | 手动 `Initialize()` + `Instance` | 全局 10+ 处 | 非线程安全初始化，隐式依赖 |

这三个 static/singleton 通过 `PathManager.Instance`、`AppLogService.Instance`、`CrashLogger.AddBreadcrumb()` 在 View、ViewModel、Service 层被广泛直接调用，形成**隐式全局依赖**。

### 3.5 MVVM 违规

#### View code-behind 包含业务逻辑

| View | 行数 | 违规内容 |
|------|------|---------|
| `ConfigCenterView.axaml.cs` | 1 317 | `new BlobServiceClient()`、`File.OpenRead()`、`PathManager.Instance` 操作 |
| `MediaStudioWindow.axaml.cs` | 1 286 | `File.Delete()`、`Directory.CreateDirectory()`、`Process.Start()`、`AppLogService.Instance` |
| `MediaStudioView.axaml.cs` | 1 277 | 几乎与 Window 版本完全重复 |
| `ConfigView.axaml.cs` | 548 | `PathManager.Instance.SessionsPath` 直接访问 |
| `ImagePreviewWindow.axaml.cs` | 379 | `File.Exists()`、`File.OpenRead()`、Bitmap 加载 |

#### Model 含业务逻辑

`AzureSpeechConfig.cs` 包含 `GetEffectiveRegion()`、`GetEffectiveEndpoint()` 等方法，并引用 `PathManager.Instance`。

#### ViewModel 间直接引用

`BatchProcessingViewModel` 直接持有 `FileLibraryViewModel` 和 `PlaybackViewModel` 的引用，未通过接口或消息总线解耦。

### 3.6 代码重复

`MediaStudioView.axaml.cs`（1 277 行）与 `MediaStudioWindow.axaml.cs`（1 286 行）包含近乎完全相同的代码——嵌入式视图和独立窗口两套实现共享了相同的业务逻辑，但未抽取公共基类或 Helper。

### 3.7 无测试基础设施

- 无 xUnit/NUnit/MSTest 项目
- 无任何自动化测试
- 由于 0 个接口 + 静态依赖 + 构造函数副作用，当前代码几乎无法编写单元测试

### 3.8 自定义 RelayCommand 冗余

项目自行实现了 `RelayCommand`（`ViewModels/RelayCommand.cs`），但已引用 `CommunityToolkit.Mvvm 8.4.0`，该库提供了功能更完善的 `RelayCommand` 和 `AsyncRelayCommand`（含自动 `CanExecuteChanged`、异常传播、取消支持等）。

## 四、改进计划

### Phase 1：接口抽取 + DI 扩展（基础层，优先级最高）

**目标：** 为所有 Service 和关键 ViewModel 定义接口，并注册到 DI 容器。

| 步骤 | 内容 | 影响范围 |
|------|------|---------|
| 1.1 | 为所有 Service 提取接口（`IConfigurationService`、`IAzureTokenProvider`、`ISpeechTranslationService`、`IAiInsightService`、`IPathManager` 等） | Services/ |
| 1.2 | 将 `PathManager`、`AppLogService`、`CrashLogger` 从 static/singleton 改为实例服务，注册为 `Singleton` | Services/、App.axaml.cs |
| 1.3 | 将所有 ViewModel 注册到 `IServiceCollection`，使用构造函数注入替代手动 `new` | ViewModels/、App.axaml.cs |
| 1.4 | 移除 `App.Services` 静态属性，改为标准构造函数注入链 | App.axaml.cs、全局 |
| 1.5 | 用接口替换 ViewModel 中的 `Func<>`/`Action<>` 回调委托 | ViewModels/ |

### Phase 2：拆分皇帝类

**目标：** 将过大的类拆分为单一职责的小类。

| 步骤 | 内容 | 预期效果 |
|------|------|---------|
| 2.1 | 从 `MainWindowViewModel` 提取 `TranslationOrchestrator` 服务（翻译启停、事件处理） | 减少约 300 行 |
| 2.2 | 从 `MainWindowViewModel` 提取 `AppLifecycleManager`（PostShow 初始化、窗口引用管理） | 减少约 100 行 |
| 2.3 | 从 `MainWindowViewModel` 提取 `EditorStateManager`（编辑器显示模式、编辑器类型切换） | 减少约 100 行 |
| 2.4 | 从 `SpeechTranslationService` 提取 `AudioCaptureManager`（设备选择、音频流管理） | 独立音频层 |
| 2.5 | 从 `SpeechTranslationService` 提取 `SubtitleWriter`（SRT/VTT 写入） | 独立输出层 |
| 2.6 | 从 `BatchProcessingViewModel` 提取批量 API 调度逻辑为 Service | 减少 ViewModel 体量 |

### Phase 3：MVVM 合规整改

**目标：** 将 View code-behind 中的业务逻辑迁移到 ViewModel/Service。

| 步骤 | 内容 |
|------|------|
| 3.1 | 将 `ConfigCenterView.axaml.cs` 中的 `BlobServiceClient` 操作、文件 I/O 移至 Service |
| 3.2 | 将 `MediaStudioView/Window` 的文件操作、`Process.Start`、日志逻辑移至 ViewModel |
| 3.3 | 合并 `MediaStudioView.axaml.cs` 和 `MediaStudioWindow.axaml.cs` 的重复代码至公共 Helper 或基类 |
| 3.4 | 将 `ImagePreviewWindow.axaml.cs` 的文件加载移至 ViewModel |
| 3.5 | 将 `AzureSpeechConfig` 中的 endpoint 计算逻辑移至专用 Service |

### Phase 4：测试基础设施

**目标：** 建立可测试的项目结构。

| 步骤 | 内容 |
|------|------|

## 六、当前架构优点

在指出问题的同时，也应认可以下做法：

1. **MVVM 基础骨架已建立** — 使用 CommunityToolkit.Mvvm 的 `ObservableObject`，ViewModelBase 继承链清晰
2. **NavigationView Shell 模式** — FluentAvalonia NavigationView + Panel/IsVisible 页面切换，符合 WinUI 惯例
3. **CrashLogger 诊断能力** — 崩溃日志包含 GC/内存/线程池/上下文快照/面包屑，信息量充足
4. **PostShow 初始化机制** — `RegisterPostShowInitializationAction` 延迟到窗口显示+配置加载后执行，避免冷启动阻塞
5. **InlineListContextMenuController** — 右键菜单已抽象为可复用控件
6. **配置保存防抖机制** — `QueueConfigSave` 避免频繁写盘
7. **CommunityToolkit.Mvvm 已引入** — 为后续迁移到 `[ObservableProperty]`/`[RelayCommand]` 提供了条件
