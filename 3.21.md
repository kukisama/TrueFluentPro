# 3.21 Media Studio Chat Markdown 渲染改造经验总结

## 一、Cherry Studio (Electron+React) 研究收获

### 架构概览
- 技术栈：Electron + React，聊天消息用 `react-markdown` + remark/rehype 插件链渲染
- 项目路径：`Docs\cherry-studio-1.7.19`

### 流式渲染：useSmoothStream
- **字符级平滑队列**：AI 返回的 token 不直接渲染，先进入 `pendingChars` 队列
- **requestAnimationFrame 驱动**：每帧从队列取若干字符追加到显示文本，实现打字机效果
- 核心逻辑：`chunkSize` 动态调节（积压多时加速、少时减速），视觉上比直接 flush 更平滑
- **关键启示**：流式体验不只是"收到就显示"，而是有一层动画缓冲层

### 代码高亮：Shiki
- Cherry Studio 用 Shiki（基于 TextMate 语法）做代码高亮，支持所有 VS Code 主题
- 比关键字匹配精确得多，但体积大、初始化慢
- 他们包了 `CodeBlock` 组件 + 虚拟滚动 `CodeViewer`，长代码不卡

### 复制机制
- 代码块右上角复制按钮，点击后 `navigator.clipboard.writeText(code)`
- 普通文本靠浏览器原生选择 + Ctrl+C
- **React VDOM diff 是关键**：重渲染时只 patch 变化节点，DOM 结构不重建 → 浏览器选区自然保留

### 映射到 Avalonia 的核心差异
| 维度 | Cherry Studio (React) | 我们 (Avalonia) |
|------|----------------------|-----------------|
| 渲染模型 | VDOM diff，只 patch 变化节点 | 无 VDOM，Children.Clear() 全量重建 |
| 文本选择 | 浏览器原生 Selection API，跨 DOM 节点 | SelectableTextBlock 只能选单个控件内文本 |
| 代码高亮 | Shiki (TextMate grammar) | 自建 CodeHighlighter (关键字匹配) |
| 流式动画 | requestAnimationFrame + 字符队列 | 80ms Dispatcher.Post 直接刷新 |

## 二、我们项目当前实现（Markdig AST + 增量渲染）

### 已完成的改造
1. **Markdig 1.1.1 引入**：替代之前的正则解析，用 AST 正确处理 CommonMark
2. **增量渲染**：比较每个 block 的源文本（`_blockSources`），只重建变化的尾部块，流式时已完成的段落不闪烁
3. **语法高亮 CodeHighlighter**：支持 20+ 语言（Python/C#/JS/TS/JSON/XML/Rust/Go 等），VS Code Dark+ 配色
4. **排版提升**：Microsoft YaHei UI 字体、14px/24px 行高、0.2 字间距、代码块圆角+header+复制按钮、表格对齐+交替行、引用左边框+背景
5. **主题适配**：`CodeInlineForegroundBrush`、`TableBorderBrush`、`CodeBlockBackgroundBrush` 等动态资源

### 架构：多 SelectableTextBlock (StackPanel)
```
ChatMarkdownBlock : StackPanel
  ├─ SelectableTextBlock  (段落1)
  ├─ SelectableTextBlock  (标题)
  ├─ Border > StackPanel  (代码块，内含 SelectableTextBlock)
  ├─ SelectableTextBlock  (段落2)
  ├─ Grid                 (表格，每个单元格独立 SelectableTextBlock)
  └─ ...
```

## 三、当前已知问题

### ⚠️ 核心问题：文本选择只能逐块进行
- **现象**：用户只能在单个 SelectableTextBlock 内拖选文字，无法从段落1一直选到代码块再到段落2
- **根因**：Avalonia 的 `SelectableTextBlock` 是独立控件，各自维护选区，不像浏览器 Selection API 能跨 DOM 节点
- **右键"全选"**：也只能全选当前右键点击的那个 STB，不是整条消息

### 尝试过的方案：单 STB + Inlines
- **思路**：把所有块渲染到一个 SelectableTextBlock 的 Inlines 集合中
- **结果**：**不可行** — Inline (Run/Span) 不支持 `Opacity`、`LetterSpacing`、`Margin` 等属性；代码块失去 Border 背景和复制按钮；表格只能降级为纯文本管道符对齐；视觉效果严重退化
- **结论**：SelectableTextBlock 的 Inlines 体系太弱，无法承载丰富排版

### 其他小问题
- 代码高亮是关键字级别，不如 Shiki/TextMate 精确（函数参数、变量作用域等无法区分）
- 流式渲染是直接 80ms flush，没有 Cherry Studio 那种字符级平滑动画

## 四、Markdig 已引入 → 可以单独做 Markdown 渲染模块

### 机会
现在项目已经依赖 Markdig 1.1.1，AST 解析能力完整。可以大张旗鼓地做一个**专门的 Markdown 渲染模块**，不局限于 Chat 场景，也可以替代目前 About/Help/Insight/Review 等页面使用的 `Markdown.Avalonia`。

### 建议的模块规划
```
Controls/Markdown/
  ├─ MarkdownRenderer.cs       — 核心：Markdig AST → Avalonia 控件树
  ├─ CodeHighlighter.cs        — 语法高亮（从 ChatMarkdownBlock 中提取）
  ├─ MarkdownTheme.cs          — 主题/样式配置（字体、颜色、间距）
  └─ SelectionManager.cs       — 跨控件选择协调器（解决核心问题）
```

### 跨控件选择的可能解法
1. **SelectionManager 方案**：在 StackPanel 层拦截 PointerPressed/PointerMoved，自行计算哪些子 STB 在选区范围内，调用各 STB 的 SelectionStart/SelectionEnd 实现视觉上的"跨块选择"，复制时拼接所有选中文本
2. **RichTextBlock 方案**：如果 Avalonia 未来支持 RichTextBlock（类似 WPF），可以在一个控件内混排 Block 和 Inline
3. **WebView 方案**：极端情况下用 WebView 渲染 HTML（Cherry Studio 本质就是这样），但引入重依赖

### 优先级建议
- P0：提取独立模块，CodeHighlighter 独立文件
- P1：实现 SelectionManager 跨块选择
- P2：字符级平滑流式动画
- P3：考虑 Shiki/TextMate 级高亮（需评估 .NET 生态可用性）
