## 0. 总纲（必须先读懂再动手）

> 这一段是工作哲学，违反任何一条都视为不合格交付。

1. **完全复刻 C# 实现，禁止加戏**
   - UI 元素、字段、命令名、参数都要与 C# 端对应。
   - 你"觉得更好"的设计 = 加戏 = 打回。
   - 不确定时，**读 C# 源代码而不是猜**。

2. **接口/数据库化，未来可剥离 UI 容器**
   - 所有状态以 SQLite 为权威源。
   - 前端 Zustand 只做"缓存 + 视图模型"，重启进程或刷新页面后能完全从 SQLite 重建。
   - 业务逻辑放后端 Rust，前端只负责展示与触发命令。

3. **关 UI 不停作业**
   - 图片/视频生成在 Rust 端 ``tokio::spawn`` 跑，UI 关闭/刷新都不能中断任务。
   - 现阶段唯一彻底停业务的方式是关闭进程。
   - 主窗口"关闭"按钮在 PR-3 起改为 hide-to-tray。

4. **刷新页面不影响作业**
   - 前端组件重新挂载时，必须能从后端拉回所有进行中任务的状态并重新订阅事件。
   - 测试方法：F5 刷新 webview，所有进行中状态必须 100% 恢复。

5. **懒加载**
   - 启动时只拉会话**元数据列表**（轻量）。
   - 选中某会话时才拉它的完整 bundle（消息、媒体引用、引文、附件）。
   - 同时最多保留 3 个完整加载的会话，超出按 LRU 卸载（对照 C# ``EnforceLoadedSessionLimit``）。

## 7. 严父审查规则（执行者请提前知晓）

提交后由审查者（我）按以下规则审：

1. 先读交付报告，对每个 ✅ 抽查证据
2. 实际拉取代码并跑一遍 PR 中所有"验收脚本场景"
3. 抽查 3-5 个 C# 字段对照，确认 schema 一字不差
4. 检查 ``git diff`` 是否包含范围外的改动（包含即打回）
5. 任何"加戏"、"顺便重构"、"基本可用"措辞 → 打回
6. 验收脚本场景只要有 1 个失败 → 打回
7. 通过则合并并启动下一 PR；不通过则带着具体打回理由重做
