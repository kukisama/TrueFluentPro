## 实施计划

目标：全面切换到 `Endpoints + ModelReference` 新结构，移除运行链路对旧字段（`AiConfig` / `MediaGenConfig` 中的旧终结点、旧模型、旧 AAD 字段）的依赖；旧配置界面和旧回退逻辑全部删除，不再维护双轨兼容。

### 第一阶段：盘点并冻结旧字段使用面

1. 明确待淘汰旧字段
   - `AiConfig.ApiEndpoint`
   - `AiConfig.ApiKey`
   - `AiConfig.ModelName`
   - `AiConfig.QuickModelName`
   - `AiConfig.SummaryModelName`
   - `AiConfig.DeploymentName`
   - `AiConfig.QuickDeploymentName`
   - `AiConfig.SummaryDeploymentName`
   - `AiConfig.AzureAuthMode`
   - `AiConfig.AzureTenantId`
   - `AiConfig.AzureClientId`
   - `MediaGenConfig.ImageApiEndpoint`
   - `MediaGenConfig.ImageApiKey`
   - `MediaGenConfig.ImageModel`
   - `MediaGenConfig.ImageProviderType`
   - `MediaGenConfig.ImageAzureAuthMode`
   - `MediaGenConfig.ImageAzureTenantId`
   - `MediaGenConfig.ImageAzureClientId`
   - `MediaGenConfig.VideoApiEndpoint`
   - `MediaGenConfig.VideoApiKey`
   - `MediaGenConfig.VideoModel`
   - `MediaGenConfig.VideoProviderType`
   - `MediaGenConfig.VideoAzureAuthMode`
   - `MediaGenConfig.VideoAzureTenantId`
   - `MediaGenConfig.VideoAzureClientId`
2. 先统一约束：新代码禁止再新增对上述字段的读取。
3. 在文档/注释中明确：运行时唯一可信来源是 `Endpoints`、`ModelReference`、以及终结点上的 `Models`。

### 第二阶段：抽出统一的新结构运行时解析器

1. 新增一个运行时解析服务，例如：
   - `IModelRuntimeResolver`
   - `ModelRuntimeResolver`
2. 服务职责：
   - 输入：`AzureSpeechConfig + ModelReference + 目标能力`
   - 输出：一个统一运行时对象，例如：
     - 终结点 URL
     - API Key
     - 认证方式
     - Tenant / Client
     - ProviderType
     - ApiVersion
     - 实际模型 ID
     - Azure 部署名（若为空则自动回退到模型 ID）
3. 所有业务链路统一只从这个解析器拿运行时配置，不再自己写 fallback 分支。
4. 解析失败直接返回明确错误，不再偷偷回退到旧字段。

### 第三阶段：改造文本 AI / 复盘链路

涉及文件（优先顺序）：
1. `ViewModels/BatchProcessingViewModel.cs`
   - 删除 `TryBuildReviewRuntimeConfig` 中“解析失败后复制旧 `AiConfig`”的分支。
   - 改为：只接受 `ModelReference` 解析成功的结果。
2. `Services/AiInsightService.cs`
   - 去掉对 `AiConfig.GetModelName()` / `GetDeploymentName()` 的依赖。
   - 改为直接接收已解析好的运行时参数。
3. `Models/AiConfig.cs`
   - 删除旧模型/部署名 fallback 方法：
     - `GetModelName()`
     - `GetDeploymentName()`
   - `IsValid` 改为只校验与新配置仍然相关的公共配置项（Prompt、开关、预设等），不再承担旧终结点完整性判断。
4. 复盘、洞察、测试连接相关调用链全部改为显式传 `ModelReference`。

### 第四阶段：改造媒体图片 / 视频链路

涉及文件（优先顺序）：
1. `ViewModels/MediaSessionViewModel.cs`
   - 删除：
     - `BuildImageAiConfig()` 中旧配置回退分支
     - `BuildVideoAiConfig()` 中旧配置回退分支
   - 改为必须通过 `ImageModelRef` / `VideoModelRef` + `Endpoints` 成功解析。
2. `ViewModels/MediaStudioViewModel.cs`
   - `TrySilentLoginForMediaAsync()` 删除 `_genConfig.ImageAzure*` / `_genConfig.VideoAzure*` 的 fallback。
   - 只根据当前引用终结点进行 AAD 静默登录。
3. `ViewModels/Settings/ImageGenSectionVM.cs`
   - 消除对 `MediaGenConfig.ImageModel` 等旧字段的依赖。
4. `ViewModels/Settings/VideoGenSectionVM.cs`
   - 消除 `SelectedVideoModel ?? Config.MediaGenConfig.VideoModel` 这种旧字段回退。
   - UI 层必须依赖 `SelectedVideoModel` 与新模型列表。
5. `Services/VideoCapabilityResolver` 相关调用点
   - 输入来源改为新选择模型，而不是旧 `VideoModel` 字符串后门。

### 第五阶段：清理导入导出与配置模型

1. `SettingsImportExportService.cs`
   - 删除：
     - `SyncAiCompatFields()`
     - `SyncMediaCompatFields()`
   - 导入后只写新结构。
2. `SettingsTransferPackage.cs`
   - 保持只导入导出新结构，不引入任何旧字段。
3. `ConfigurationService.MigrateEndpoints()`
   - 判断是否仍需保留：
     - 若仍要兼容历史 `config.json` 启动迁移，可保留一次性迁移逻辑
     - 若决定彻底放弃旧本地配置兼容，可删除
4. `AiConfig` / `MediaGenConfig`
   - 删除所有仅用于旧终结点、旧模型、旧 AAD 回退的字段
   - 保留仍属于业务配置本身的字段：Prompt、默认参数、开关、预设等

### 第六阶段：收口设置页与错误提示

1. 设置页保存时增加强校验：
   - 文本模型引用不能为空（至少满足洞察/复盘需要）
   - 图片模型引用不能为空（若启用图片能力）
   - 视频模型引用不能为空（若启用视频能力）
2. 所有业务入口遇到引用失效时：
   - 明确提示“模型引用无效/终结点不存在/模型未配置能力”
   - 不再静默回退旧配置
3. 统一错误体验：
   - UI 上报错明确指出缺的是哪一个终结点 / 哪一个模型引用

### 第七阶段：删除旧字段后的收尾

1. 全仓搜索并清除所有对旧字段的读取。
2. 删除不再需要的兼容注释、迁移桥接代码和 fallback helper。
3. 更新文档：
   - 说明配置中心现已完全基于新结构
   - 不再支持旧终结点字段
4. 再次执行一次全量静态检查与构建验证。

## 如何测试

你说你可以自己测，所以这里按“手工验收清单”来写。

### A. 设置页保存与重载

1. 新建 2 个 AI 终结点：
   - 一个文本/复盘模型终结点
   - 一个图片/视频终结点
2. 给各自添加模型，并设置能力标签。
3. 在设置页中分别选择：
   - 洞察模型
   - 复盘模型
   - 图片模型
   - 视频模型
4. 关闭应用重新打开，确认：
   - 终结点仍存在
   - 模型引用仍正确
   - 不依赖旧字段也能正确恢复

### B. 洞察 / 复盘链路

1. 打开实时洞察或复盘功能。
2. 验证文本请求能正常发出。
3. 分别测试：
   - OpenAI Compatible + API Key
   - Azure OpenAI + API Key
   - Azure OpenAI + AAD
4. 人工确认：
   - 实际调用的模型与设置页选择一致
   - Azure 情况下实际使用的是部署名，不是旧模型名 fallback

### C. 图片 / 视频链路

1. 在 Media Studio 中分别测试图片生成与视频生成。
2. 验证以下场景：
   - 新建会话后直接生成
   - 切换模型后再生成
   - AAD 登录后重新打开再生成
3. 人工确认：
   - 不存在“新引用失效却偷偷走旧 endpoint”的情况
   - 图片与视频都只依赖 `ModelReference + Endpoints`

### D. 故障场景验证

1. 手动删除某个被引用的终结点。
2. 再次进入相关功能，预期：
   - 应明确报错“引用失效”
   - 不允许悄悄回退到旧配置
3. 手动删除模型或把模型能力改错，预期：
   - 直接提示配置无效
   - 不出现误用其它模型的情况

### E. 导入导出验证

1. 导出一份新的资源配置 JSON。
2. 删除当前配置中的终结点与模型引用。
3. 再导入刚导出的 JSON。
4. 验证：
   - 所有模型引用、终结点、存储、Speech 订阅都能恢复
   - 不需要任何旧字段同步，功能仍全部可用

### F. 回归验证

每完成一个阶段，都至少执行一次：
- `dotnet build -p:UseAppHost=false`

最终验收建议至少覆盖：
- 设置页保存/重启恢复
- 洞察一次
- 复盘一次
- 图片生成一次
- 视频生成一次
- AAD 场景一次

如果以上全部通过，就说明“纯新结构运行”基本站住了。
