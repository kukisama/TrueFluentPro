# 多云资源中转网关 —— 架构设计说明

> 本文档回答 issue 中的问题：
>
> > "我希望有一个独立的后台服务，负责 Azure 很多资源中转，同时前台不知道。前台统一一个翻译方法，后台可以挂微软翻译也可以挂腾讯翻译。前台调用只知道是一种资源……前台不知道 key，通过 AAD 认证拿到身份后访问资源；资源和管理身份的 AAD 不在一起，因为后端挂的不一定是微软资源。"
>
> 这种架构的标准名称、由哪些经典模式组合而成、每个模式各自负责什么、以及在 `TrueFluentPro` 仓库里应当落到哪些文件，都在本文里详细说明。

---

## 1. 这是一种什么架构？（一句话定义）

> **一个以 BFF（Backend-for-Frontend）形态部署的 AI / 云资源网关，前端通过 AAD 委派身份接入，网关本身作为"受信子系统（Trusted Subsystem）"统一持有下游资源凭据，并在内部以"策略 + 适配器"模式抽象多家云供应商。**

业界对这种架构常用的 **几种等价叫法**：

| 叫法 | 强调的侧面 |
| --- | --- |
| **AI Gateway** / **LLM Gateway** | 工程界口语，类似 LiteLLM、Portkey、One-API、Azure APIM AI Gateway |
| **Provider-Agnostic Gateway**（厂商无关网关） | 强调"前端不感知供应商" |
| **Backend For Frontend (BFF) + Trusted Subsystem** | 微软官方架构文档用语，强调身份与凭据隔离 |
| **Anti-Corruption Layer (ACL)**（DDD） | 强调"外部领域模型不污染内部模型" |
| **Cloud Resource Broker** | 强调"凭据中介" |

这些叫法描述的是 **同一种东西的不同侧面**。在你的场景下，最完整、最贴切的一种正式表述就是上面那段加粗的话。

## 2. 这套架构是由哪些经典模式拼出来的？

整套设计可以拆成 **两层正交的关注点**：

```
┌──────────────────────────────────────────────────────────────┐
│                  第一层：前端面向的统一抽象                 │
│  (Facade + Strategy + Adapter + ACL + BFF/API Gateway)       │
├──────────────────────────────────────────────────────────────┤
│                  第二层：身份与凭据的隔离                   │
│   (Trusted Subsystem + Credential Broker + Delegated Auth)   │
└──────────────────────────────────────────────────────────────┘
```

### 第一层：前端面向的统一抽象（"前台只看到一种资源"）

| 模式 | 在你的场景里干什么 |
| --- | --- |
| **Facade Pattern（门面模式）** | 前台只看到 `POST /api/v1/translate`、`POST /api/v1/chat` 这种 **统一接口**，不暴露任何 Azure / 腾讯字样 |
| **Strategy Pattern（策略模式）** | 后台定义 `ITranslator` 接口，提供 `AzureTranslator`、`TencentTranslator` 等 **可互换的实现**；运行时按租户/灰度/配额选择一个 |
| **Adapter Pattern（适配器模式）** | 把每家厂商的 SDK / REST 响应 **适配** 成你内部统一的契约（`TranslationResult`、`ChatChunk` 等） |
| **Anti-Corruption Layer（DDD 防腐层）** | 内部业务模型 **永不**直接引用 `Azure.AI.OpenAI` 或 `TencentCloud.Tmt` 的 DTO；外部模型只在 Adapter 边界存在 |
| **BFF / API Gateway** | 整个后台 **就是前端的唯一对话方**；前端永远只跟你的域名打交道 |
| **Provider Registry / Plugin Pattern** | DI 容器里 **注册一组实现**，由工厂 `ITranslatorFactory.Resolve(tenantId)` 选出当前要用的那一个 |

### 第二层：身份与凭据的隔离（"前台不知道 key，AAD 只管身份"）

| 模式 / 概念 | 在你的场景里干什么 |
| --- | --- |
| **Delegated Identity（委派身份）** | 前端用 MSAL 登录 AAD，拿到的 access token 只**证明"我是谁"**，**不包含**任何资源密钥 |
| **Trusted Subsystem Pattern**（微软官方术语） | 下游资源信任的是 **"你的后台这个服务身份"**，而不是终端用户。后台再根据 AAD 身份做自己的授权（RBAC/ABAC） |
| **Credential Broker / Secret Zero Isolation** | 真正的密钥（Azure Key、腾讯 SecretId/SecretKey、OpenAI Key 等）**只存在于后台**：环境变量 → 加密 DB → KeyVault 三级凭据链 |
| **On-Behalf-Of (OBO) Flow** | 当下游资源**也在 Azure**且需要"以用户身份"访问时，后台用 OBO 把用户 token 换成下游 API 的 token |
| **Workload Identity Federation** | 当下游资源**在 AWS / GCP**时，AAD 颁发的 token 通过联邦换成对方云的临时凭据 |
| **Trusted Subsystem（针对非微软资源）** | 下游是**腾讯云 / 自建服务**时，没有联邦可用，就由后台直接以自己的服务凭据调用，**身份层和资源层物理上彻底分开** |

> ⚠️ 关键区分：**身份平面** 和 **资源平面** **不必属于同一个 AAD、甚至不必属于同一朵云**。这是你这个问题里最容易被忽略却最关键的一点 —— 它正是 *Trusted Subsystem* 的核心价值。

## 3. 端到端调用时序

```
┌──────────┐   ①AAD 登录          ┌────────────┐
│  前 端   │ ───────────────────▶ │   Azure AD │
│ (MSAL)   │ ◀─── access_token ── │            │
└────┬─────┘    aud=你的后台 API  └────────────┘
     │
     │ ②业务请求 + Authorization: Bearer <user token>
     ▼
┌──────────────────────────────────────────────────────────────┐
│             你的后台 (BFF / AI Gateway)                      │
│                                                              │
│  ③ JWT 验签 + 解析用户身份 (UserId / TenantId / Scopes)     │
│  ④ 业务授权（用户能否调用此能力？是否超配额？）            │
│  ⑤ ITranslatorFactory.Resolve(tenantId)                      │
│       └─▶ 返回 AzureTranslator 或 TencentTranslator          │
│  ⑥ 从 KeyVault / 加密 DB 取出对应供应商的 真实密钥          │
│  ⑦ 调用下游 (Azure / 腾讯 / 自建)，把响应适配成统一模型     │
│  ⑧ 写审计 / 计费 / 配额                                     │
│  ⑨ 把统一模型返回给前端                                     │
└──────────────────────────────────────────────────────────────┘
```

整个过程中：

- 前端 **从未** 接触任何下游资源的密钥。
- 前端 **从未** 知道实际服务方是 Azure 还是腾讯。
- AAD **只负责回答"你是谁"**，不参与"凭据怎么拿"。
- 下游资源 **只信任后台的服务身份**，对终端用户一无所知。

## 4. 落到 `TrueFluentPro` 仓库的具体映射

仓库当前已经具备的部分 ✅，以及还缺什么 ❌：

| 架构元素 | 仓库现状 | 文件 / 位置 |
| --- | --- | --- |
| BFF / 网关进程 | ✅ 已有独立的 ASP.NET Core 10 Minimal API | `backend/TrueFluentPro.Api/` |
| 委派身份接入 (AAD) | ✅ 已有 MSAL.NET 客户端 + 后台 JWT 验证 | `Services/Cloud/CloudAuthService.cs`、`backend/TrueFluentPro.Api/Program.cs` |
| 凭据中介 (Credential Broker) | ✅ env → DB(AES) → KeyVault 三级链 | `backend/TrueFluentPro.Api/` 凭据解析逻辑 |
| 前端→后台单一通道 | ✅ `ICloudApiClient` 客户端代理 | `Services/Cloud/CloudApiClient.cs` |
| 自托管 / 云模式切换 | ✅ `IServiceModeManager` (SelfHosted / Cloud) | `Services/Cloud/ServiceModeManager.cs` |
| 能力分端点 | ✅ `/api/v1/chat`、`/api/v1/speech`、`/api/v1/images`、`/api/v1/videos`、`/wss/v1/translate` | `backend/TrueFluentPro.Api/Endpoints/` |
| **显式的 Provider 抽象**（多供应商可插拔） | ❌ **目前按"能力"分了端点，但没有按"供应商"分实现**。要挂腾讯翻译就必须加这层 | 建议：`backend/TrueFluentPro.Api/Providers/` |
| **Provider 工厂 / 路由策略**（按租户/配额/灰度选 Provider） | ❌ 待补 | 建议：`ITranslatorFactory`、`IChatProviderFactory` 等 |
| 审计 / 配额（已有 `QuotaMiddleware`） | ✅ | `backend/TrueFluentPro.Api/` |

### 建议的最小演进 —— 引入 `ITranslator` + 工厂

> 下面是 **概念骨架**，目的是把"如果将来要挂腾讯翻译，应该改哪里"这件事画清楚。**不要求现在就实现**。

```csharp
// backend/TrueFluentPro.Api/Providers/Translation/ITranslator.cs
namespace TrueFluentPro.Api.Providers.Translation;

public interface ITranslator
{
    string ProviderId { get; }            // "azure" | "tencent" | ...
    Task<TranslationResult> TranslateAsync(
        TranslationRequest request,
        CancellationToken ct);
}

public sealed record TranslationRequest(
    string Text, string FromLang, string ToLang, string TenantId);

public sealed record TranslationResult(
    string TranslatedText, string DetectedFrom, string ProviderId);
```

```csharp
// backend/TrueFluentPro.Api/Providers/Translation/AzureTranslator.cs
internal sealed class AzureTranslator : ITranslator
{
    public string ProviderId => "azure";
    // 内部：用 KeyVault 里的 Azure Translator key 调 Azure
}

// backend/TrueFluentPro.Api/Providers/Translation/TencentTranslator.cs
internal sealed class TencentTranslator : ITranslator
{
    public string ProviderId => "tencent";
    // 内部：用 KeyVault 里的腾讯 SecretId/SecretKey 调 TMT
}
```

```csharp
// backend/TrueFluentPro.Api/Providers/Translation/ITranslatorFactory.cs
public interface ITranslatorFactory
{
    /// <summary>按租户 / 配额 / 灰度策略选择一个 Provider。</summary>
    ITranslator Resolve(string tenantId);
}
```

```csharp
// Program.cs 注册
builder.Services.AddSingleton<ITranslator, AzureTranslator>();
builder.Services.AddSingleton<ITranslator, TencentTranslator>();
builder.Services.AddSingleton<ITranslatorFactory, TranslatorFactory>();

// 端点保持不变，前端永远只看到这一个 URL
app.MapPost("/api/v1/translate", async (
    TranslationRequest req,
    ClaimsPrincipal user,
    ITranslatorFactory factory,
    CancellationToken ct) =>
{
    var tenantId = user.FindFirstValue("tid")!;
    var translator = factory.Resolve(tenantId);
    var result = await translator.TranslateAsync(req with { TenantId = tenantId }, ct);
    return Results.Ok(result);
});
```

> 同样的骨架可以套用到 `IChatProvider`（Azure OpenAI / DeepSeek / 通义千问 / 腾讯混元）、`ISpeechProvider`、`IImageProvider`、`IVideoProvider`。

## 5. 常见疑问 FAQ

**Q1：Facade 和 BFF 是不是一回事？**
不是。Facade 是 **代码层** 的"用一个类把多个子系统裹起来"；BFF 是 **部署层** 的"专门为某种前端形态做的后端服务"。在你的场景里二者叠加：BFF 这个进程内部，又用 Facade 把多 Provider 包装成统一 API。

**Q2：为什么不让前端直接拿 AAD token 去调 Azure 资源？**
- 一旦腾讯 / 自建资源进来，AAD token 在那里 **没有意义**；
- 前端持有任何"能直接花钱"的凭据都会带来配额、审计、撤销的难题；
- 多 Provider 抽象、统一计费、灰度发布都依赖"流量必须经过你的后台"。

所以正确做法是：**AAD 只用于"前端 ↔ 你的后台"这一段**；后台到下游是另一套独立的凭据体系。

**Q3：AAD 不是只能管 Azure 吗？怎么管腾讯？**
AAD 在这里 **不管腾讯资源**，它只回答"调用我后台的这个人是谁"。腾讯资源由后台 **以自己的服务身份**访问，再根据 AAD 给的"是谁"做内部授权决策（这正是 Trusted Subsystem）。

**Q4：什么时候用 OBO，什么时候用 Trusted Subsystem？**
- 下游 **也在 Azure** 且需要做"按用户级别的细粒度授权"（如让用户只能看自己的 SharePoint 文件） → **OBO**。
- 下游是 **非 Azure** 资源，或者授权决策完全在你自己的业务里做 → **Trusted Subsystem**。
- 二者可以共存：同一个后台对 Azure 走 OBO、对腾讯走 Trusted Subsystem。

**Q5：这套东西有现成的开源项目可参考吗？**
- **LiteLLM**、**Portkey**、**One-API**：典型的 LLM Gateway，多 Provider 抽象 + 统一接口
- **Azure API Management + AI Gateway 策略**：微软官方的 LLM Gateway 方案
- **Kong / Envoy**：通用 API 网关，可作为 BFF 之前的另一层

---

## 6. 一句话总结

> 你想要的，叫做 **"用 AAD 做委派身份、用 Trusted Subsystem 做凭据隔离、用 Strategy + Adapter 做多 Provider 抽象、对外以 BFF / API Gateway 形态暴露统一接口的多云 AI 网关"**。
>
> 现仓库 `backend/TrueFluentPro.Api` 已经把 **BFF + 委派身份 + 凭据中介** 三件事做完了；下一步只需要在 `Providers/` 下加一层 **`I*Provider` + `*ProviderFactory`**，就能正式具备"同一个翻译端点，后台想挂哪家挂哪家"的能力。
