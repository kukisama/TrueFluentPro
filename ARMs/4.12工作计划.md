# 听析中心架构重构 — 任务队列化工作计划

> 日期：2026-04-12
> 状态：设计评审

---

## 一、核心目标

将听析中心从当前的 **ViewModel 直接调度+执行+展示** 三合一模式，重构为 **任务队列 + DB 状态中转 + 事件通知** 的前后端分离架构。

**关键收益**：
- 消除 `IsGenerating` 单标志互斥导致的按钮全锁死问题
- 消除共享 CancellationToken 导致的"停一个全停"问题
- 每个任务独立生命周期，独立取消、独立失败重试
- DB 为唯一状态真相源，客户端只提交任务 + 读取结果
- 架构天然支持未来云端前后端分离

## 三、任务标识设计

### 3.1 唯一标识方案

每个任务用 **`task_id`** 作为全局唯一标识，格式：

```
{ULID}
```

选用 ULID（Universally Unique Lexicographically Sortable Identifier）而非 GUID：
- **可排序**：前 10 字符是时间戳（毫秒），天然按提交时间排序
- **唯一性**：后 16 字符随机，碰撞概率极低
- **可读性**：26 字符，比 GUID 短，纯大写字母+数字
- **示例**：`01HZXK3V7P4R8Q2NMJW6TGBYFS`

### 3.2 任务复合键

同一音频的同一阶段不应重复排队。用 `(audio_item_id, stage)` 作为业务去重键：
- 提交任务前检查：该组合是否已有 Pending/Running 的任务
- 有则拒绝或返回已有 task_id
- 这保证了队列中不会出现同一音频的同一阶段多个并行任务

## 四、任务状态机

```
                   提交
                    │
                    ▼
              ┌──────────┐
              │  Pending  │ ← 等待前置依赖完成 / 等待并发槽
              └────┬─────┘
                   │ 依赖满足 + 有空闲槽
                   ▼
              ┌──────────┐
              │  Running  │ ← 正在执行
              └──┬────┬──┘
                 │    │
        成功完成  │    │ 异常/超时
                 ▼    ▼
           ┌─────┐  ┌────────┐
           │Done │  │Failed  │
           └─────┘  └───┬────┘
                        │ 用户点击重试
                        ▼
                  ┌──────────┐
                  │  Pending  │ (重新入队)
                  └──────────┘

  任何 Pending/Running 状态均可被用户取消 → Cancelled
```

### 状态枚举

```csharp
public enum AudioTaskStatus
{
    Pending,      // 已提交，等待调度
    Running,      // 执行中
    Completed,    // 已完成，结果已写入 audio_lifecycle
    Failed,       // 执行失败，error_message 记录原因
    Cancelled,    // 用户主动取消
}
```

## 五、数据库设计

### 5.1 新增 `audio_task_queue` 表

```sql
CREATE TABLE IF NOT EXISTS audio_task_queue (
    task_id         TEXT    NOT NULL PRIMARY KEY,     -- ULID，全局唯一
    audio_item_id   TEXT    NOT NULL,                 -- FK → audio_library_items.id
    stage           TEXT    NOT NULL,                 -- AudioLifecycleStage 枚举值
    status          TEXT    NOT NULL DEFAULT 'Pending', -- AudioTaskStatus
    priority        INTEGER NOT NULL DEFAULT 0,       -- 优先级（0=普通，数字越大越优先）
    depends_on      TEXT,                             -- 前置 task_id（JSON 数组），null = 无依赖
    error_message   TEXT,                             -- Failed 时记录错误信息
    retry_count     INTEGER NOT NULL DEFAULT 0,       -- 已重试次数
    submitted_at    TEXT    NOT NULL,                 -- ISO 8601 提交时间
    started_at      TEXT,                             -- 开始执行时间
    completed_at    TEXT,                             -- 完成/失败时间
    submitted_by    TEXT,                             -- 提交者标识（未来多用户场景）

    FOREIGN KEY (audio_item_id) REFERENCES audio_library_items(id)
);

-- 高频查询索引
CREATE INDEX IF NOT EXISTS idx_task_queue_status
    ON audio_task_queue(status);
CREATE INDEX IF NOT EXISTS idx_task_queue_audio_stage
    ON audio_task_queue(audio_item_id, stage, status);
```

### 5.2 现有表不变

- `audio_library_items` — 不变
- `audio_lifecycle` — 不变，仍作为生成结果的持久存储

**关系**：`audio_task_queue` 是"任务调度"，`audio_lifecycle` 是"最终产出"。任务完成后写 `audio_lifecycle`，队列记录保留用于审计/监控。

## 六、DAG 依赖定义

```csharp
public static class AudioTaskDependencies
{
    /// <summary>
    /// 定义每个阶段的前置依赖阶段。
    /// 前置阶段的 audio_lifecycle 必须存在且 is_stale = 0 才可开始。
    /// </summary>
    public static readonly IReadOnlyDictionary<AudioLifecycleStage, AudioLifecycleStage[]> Prerequisites
        = new Dictionary<AudioLifecycleStage, AudioLifecycleStage[]>
    {
        [AudioLifecycleStage.Transcribed]   = Array.Empty<AudioLifecycleStage>(),  // 无依赖
        [AudioLifecycleStage.Summarized]    = new[] { AudioLifecycleStage.Transcribed },
        [AudioLifecycleStage.MindMap]       = new[] { AudioLifecycleStage.Summarized },
        [AudioLifecycleStage.Insight]       = new[] { AudioLifecycleStage.Transcribed },
        [AudioLifecycleStage.PodcastScript] = new[] { AudioLifecycleStage.Transcribed },
        [AudioLifecycleStage.PodcastAudio]  = new[] { AudioLifecycleStage.PodcastScript },
        [AudioLifecycleStage.Research]      = new[] { AudioLifecycleStage.Transcribed },
        [AudioLifecycleStage.Translated]    = new[] { AudioLifecycleStage.Transcribed },
    };
}
```

### 自动级联提交

当用户打开一个新音频，系统自动检查 DB 中缺失的阶段，按 DAG 顺序批量提交任务：

```
缺 Transcribed → 提交转录任务
缺 Summarized  → 提交总结任务 (depends_on: 转录 task_id)
缺 MindMap      → 提交脑图任务 (depends_on: 总结 task_id)
缺 Insight      → 提交顿悟任务 (depends_on: 转录 task_id)
缺 PodcastScript→ 提交播客任务 (depends_on: 转录 task_id)
缺 Research     → 提交研究任务 (depends_on: 转录 task_id)
```

## 七、核心接口设计

### 7.1 事件总线

```csharp
public interface ITaskEventBus
{
    /// <summary>任务状态变更事件</summary>
    event Action<TaskStatusChangedEvent>? TaskStatusChanged;

    void Publish(TaskStatusChangedEvent e);
}

public record TaskStatusChangedEvent(
    string TaskId,
    string AudioItemId,
    AudioLifecycleStage Stage,
    AudioTaskStatus OldStatus,
    AudioTaskStatus NewStatus,
    string? ErrorMessage = null
);
```

本地实现：内存 `event`，UI 线程 Dispatch。
未来云端：替换为 SignalR client 接收 → 转发到同一接口。

### 7.2 任务队列服务

```csharp
public interface IAudioTaskQueueService
{
    /// <summary>提交单个任务，返回 task_id。若已存在 Pending/Running 的同 (audio, stage) 任务则返回已有 task_id。</summary>
    string Submit(string audioItemId, AudioLifecycleStage stage, int priority = 0);

    /// <summary>为音频自动提交所有缺失阶段的任务（按 DAG 依赖）</summary>
    List<string> SubmitAll(string audioItemId);

    /// <summary>取消指定任务</summary>
    void Cancel(string taskId);

    /// <summary>取消指定音频的所有 Pending/Running 任务</summary>
    void CancelAllForAudio(string audioItemId);

    /// <summary>重试失败的任务（重置为 Pending）</summary>
    string Retry(string taskId);

    /// <summary>查询任务列表（支持筛选）</summary>
    List<AudioTaskRecord> Query(AudioTaskQueryFilter? filter = null);
}
```

### 7.3 任务执行器（后台调度）

```csharp
public interface IAudioTaskExecutor
{
    /// <summary>启动后台调度循环</summary>
    Task StartAsync(CancellationToken appShutdown);

    /// <summary>通知有新任务入队，唤醒调度循环</summary>
    void NotifyNewTask();
}
```

调度循环伪代码：
```
loop:
    等待信号（新任务入队 / 某任务完成释放槽位）
    从 DB 取下一个可执行任务：
        status = Pending
        AND 前置依赖全部 Completed（查 audio_lifecycle 非 stale）
        AND 并发槽未满
    若有 → 标记 Running → 在线程池执行 → 完成后写 lifecycle + 更新 queue → 发事件
```

## 九、对 AudioLabView 的改造

### 改造前（当前）
```
用户打开音频 → ViewModel 直接调用 TranscribeSessionAsync()
             → 内存中 IsGenerating = true，全按钮禁用
             → 流式写内存 → PostIfActive 推 UI
             → 完成写 DB → IsGenerating = false
```

### 改造后
```
用户打开音频 → ViewModel 调用 taskQueue.SubmitAll(audioItemId)
             → 从 DB 读取各阶段状态，显示"排队中/生成中/已完成"
             → 订阅 TaskStatusChanged 事件
             → 事件到达 → 从 DB 读取最新结果 → 刷新对应 Tab 内容
```

**AudioLabViewModel 改造要点**：
- 删除所有 `GenerateXxxSessionAsync()` 方法（移入 TaskExecutor）
- 删除 `IsGenerating`、`ResetCts()`、`PostIfActive()` 等流式推送机制
- 每个 Tab 的状态变为三态：`Empty`（无数据）/ `Processing`（有 Pending/Running 任务）/ `Ready`（有 lifecycle 数据）
- Tab 切换时从 DB 读取对应阶段内容，不依赖内存缓存
- 保留 `AudioFileSession` 但大幅精简，仅存 UI 层面的临时编辑状态

## 十一、风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| SQLite 并发写入（WAL 模式下仍有写锁） | TaskExecutor 内部串行写 DB，读可并发 |
| AI API 限流 | 信号量控制并发上限（默认 2），可配置 |
| 应用崩溃后队列中 Running 的任务 | 启动时扫描 Running 状态任务，超过一定时间未完成的重置为 Pending |
| 重构过程中功能回归 | Phase 3-4 逐个阶段迁移，每迁一个验证一个 |
